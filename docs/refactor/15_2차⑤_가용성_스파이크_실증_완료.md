# 15. 2차 리팩토링 ⑤ 가용성 스파이크 실증 — 완료

- **완료일**: 2026-07-23
- **설계 문서**: [⑤ 가용성 스파이크 벤치마크 설계](../superpowers/specs/2026-07-22-availability-spike-benchmark-design.md)
- **측정 계획**: [⑤ 가용성 스파이크 벤치마크 측정 계획](../superpowers/plans/2026-07-22-availability-spike-benchmark.md)
- **벤치마크 문서**: [23번 벤치마크 — A/B 판정표 + 원본 출력](../benchmarks/23-2026-07-23-availability-spike-ab.md)

## 왜 필요했나

①(접수-매칭 분리)·②(자금 홀드 그룹커밋)·③(취소 우선 스케줄링)·④(입장 거절 품질)는
전부 "실증은 ⑤"로 수치를 미뤘다. 22번 스파이크가 노출한 두 실패(취소 43.9% 실패,
접수 p95 14.8s)를 이 네 조각이 실제로 고쳤는지, 더 높은 목표 부하(5,000 hold +
~10,000 초과 버스트)에서 same-session A/B로 실증하는 것이 ⑤의 목적이었다.

## 어떻게 검증했나

19~22번과 동일한 same-session A/B 방법론: A(`c899284`, 2차 이전)를 배포·측정 →
리셋 → B(`5a5c7f1`≈`daf9d3e`, ①②③④ 전부 반영) 배포·측정, 부하 프로파일
`300 웜 → 5,000 램프 → 5,000 hold(3분) → ~10,000 버스트(45초) → 300 급락 →
회복(2분)`. 22번의 `order-spike-single-symbol.js`를 계승해
`_workspace/loadtest/order-spike-availability.js`(리포 커밋)로 확장했다 — 503
응답 분류, `Retry-After` 준수 여부 추적을 추가.

측정 중 인프라 이슈(로컬 도구 세션 누적 지연, load-gen VM이 VU 10,000 근처에서
반복 응답 불능)를 겪었으나 커널 로그로 애플리케이션 무관임을 확인하고 대응했다
(자세한 내용은 23번 문서의 "실측 함정" 절). B는 burst 램프 ~98% 지점에서
load-gen이 재차 죽어 급락·회복 단계에 도달하지 못했고, 사용자 확인 하에
크래시 직전 데이터로 판정했다.

## 결과 — ①②③④의 미뤄둔 수치가 채워졌다

**①②④가 만드는 하드 보장("매달림 0, 항상 바운디드 응답")은 확실히 실증됐다.**
A는 hold·버스트에서 초 단위(p95 24~29초, 회복 후 80초까지도 미완전 회복)로
매달렸다. B는 hold·버스트 내내 성공이든 거절(503)이든 항상 밀리초 단위로
응답했다 — 초 단위 매달림이 단 한 건도 없었다. 접수 p95는 hold(5,000)에서
초(24.06s, A) → ms(130.66ms, B 성공분 기준)로 자릿수가 개선됐다.

**처리량 천장은 목표(5,000 동시 제출자)에 못 미쳤다.** hold에서 B의 셰딩률은
93.1% — 실제 처리 가능한 천장이 5,000보다 훨씬 아래라는 뜻이다. 이는 설계
문서의 "판정 실패 시" 분기가 정확히 예견한 시나리오("5,000에서 대량 셰딩:
천장이 5,000 아래 — 응답이 바운디드면 가용성 정의상 합격이나, 목표 미달이면
후속 승격 근거")다. 다만 병목이 DB(21·22번의 결론)에서 **엔진 자체 처리량**
으로 옮겨간 것 자체가 ②가 DB 왕복 문제를 실제로 풀었다는 증거이기도 하다.

**취소 실패(P1 vs P2)가 실측으로 명확히 갈렸다.** ③이 고친 것은 select
불공정(P1)이고, 이번에 지배적으로 관측된 취소 실패는 엔진이 `ExecutionCh`에서
블록되는 P2다 — ③의 설계 문서가 애초에 범위 밖(①②④의 몫)이라 명시한 부분과
정확히 일치한다. 채널 게이지(`cancel`=10~36 여유, `execution`=1024 포화)가
이를 직접 증명했다 — P1은 실제로 사라졌으나(취소 채널 자체는 절대 포화되지
않음) P2가 지배적이라 체감 실패율 감소로 이어지지 않았다.

**정합성은 두 arm 모두 5항목 전부 0** — 취소가 대량 실패해도(A 52.53%, B
48.67%) 자금·상태 정합성은 전혀 깨지지 않았다. A-3/A-4/A-6 보증 체인이 목표
초과 부하에서도 그대로 버텼다.

자세한 수치·판정표·해석은 [23번 벤치마크 문서](../benchmarks/23-2026-07-23-availability-spike-ab.md)
참고.

## 이 조각의 성격

**코드 변경 없음 — 측정과 문서화만.** k6 스크립트(`_workspace/loadtest/order-spike-availability.js`)
1건만 리포에 커밋됐다.

## 남은 것 / 후속

- **엔진 자체 처리량/`ExecutionCh` 다운스트림 병목**: hold(5,000)에서 93.1%
  셰딩의 다음 병목 후보. pprof 등으로 정확한 지점 확인이 다음 단계.
- **취소 P2(엔진 `ExecutionCh` 정지) 완화**: ③은 P1만 고쳤다 — 후속 브레인스토밍
  대상.
- **샤딩 코디네이터 vs 엔진 샤딩 자체 확대**: 위 병목 확인 후 측정 기반 판단.
- **load-gen VM 10,000 VU 근처 불안정성**: 다음 대규모 부하 세션 착수 전
  우선 조사 또는 완화(VU 상한 하향) 권고.

이로써 **2차 리팩토링(가용성 100%)이 코드+실증까지 완결**됐다. ①②③④의 코드는
전부 완성됐고, ⑤가 처음으로 그 효과(하드 보장 실증)와 한계(처리량 천장 미달,
취소 P2)를 동시에 정직하게 드러냈다 — 다음 사이클의 구체적 출발점이 됐다.
