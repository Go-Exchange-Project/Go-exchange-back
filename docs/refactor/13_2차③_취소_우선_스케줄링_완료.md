# 13. 2차 리팩토링 ③ 취소 우선 스케줄링 — 완료

- **완료일**: 2026-07-21
- **설계 문서**: [③ 취소 우선 스케줄링 설계](../superpowers/specs/2026-07-20-cancel-priority-scheduling-design.md)
- **구현 계획**: [③ 취소 우선 스케줄링 구현 계획](../superpowers/plans/2026-07-20-cancel-priority-scheduling.md)
- **근거**: [22번 벤치마크](../benchmarks/22-2026-07-18-outbox-ab-and-spike.md) — 급등 스파이크에서 취소 시도의 43.9%(7,959건)가 500 실패(`CancelCh` 만석 → 1초 타임아웃).

## 왜 문제였나

`MatchingEngine.Start`의 엔진 루프는 `OrderCh`·`CancelCh`·ticker·`stopCh`를 **동등
우선순위 `select`**로 처리했다. Go의 `select`는 준비된 case 중 무작위로 선택하므로,
주문 부하가 높으면 취소는 주문과 ~50:50으로 경쟁 → 지속 부하에서 취소가 굶는다 →
`CancelCh` 만석 → `OrderService.CancelOrder`의 1초 send-timeout 발동 → 500.

취소 실패는 두 원인으로 나뉜다: **P1(select 불공정)** — 엔진이 살아있어도 동등
select라 취소가 주문과 경쟁해 굶는 것. **P2(ExecutionCh 정지)** — 엔진이
`emitTrade`의 블로킹 송신에서 멈추면 select 자체가 안 돌아 취소가 아예 처리되지
않는 것. ③은 **P1만** 구조적으로 없앤다. P2는 ①(주문 게이트)·②(DB 천장 상향으로
엔진이 덜 멈춤)·④(목표 초과 시 빠른 거절)의 몫이다.

오더북은 단일 writer(엔진 goroutine이 락 없이 소유)라 취소(`RemoveOrder`)는 반드시
엔진 goroutine에서 실행돼야 한다 — 별도 스레드로 분리가 불가능하므로, **goroutine
내 우선순위(priority select)** 가 P1을 없애는 유일한 구조적 방법이다.

## 어떻게 해결했나 (Task 1, 커밋 `3b9e88d`)

`MatchingEngine.Start`의 select **한 곳만** 2단계 priority select로 교체했다:

1. **1단계**: `CancelCh`를 논블로킹(`select`+`default`)으로 확인. 취소가 있으면
   처리 후 `continue`로 1단계를 다시 확인 — 대기 중인 취소를 **전부 드레인**한
   뒤에야 2단계로 넘어간다.
2. **2단계**: 취소가 없을 때만 기존 블로킹 select(`CancelCh`/`OrderCh`/ticker/
   `stopCh`) — 1단계의 `default` 직후 도착한 취소를 놓치지 않도록 `CancelCh`
   case를 그대로 남겨둔다.

`processOrder`/`processCancel`/`drainPendingWork`/`emit*`/shutdown 도미노(드레인 →
스냅샷 flush → `ExecutionCh`/`SnapshotCh` close)는 전부 무변경 — 변경은 스케줄링
순서뿐이다. `CancelCh` 버퍼(1024)·`OrderService.CancelOrder`의 1초 send-timeout·
`ShardedEngine`도 무변경(각 샤드가 이 엔진을 그대로 쓰므로 자동 커버).

## 결과

- **우선순위 결정론적 증명**: 오더북에 취소 대상(price 200 매도 5건)과 매칭 대상
  (price 100 매도 5건)을 가격 레벨로 분리 시드하고, `CancelCh`에 취소 5건·
  `OrderCh`에 체결을 유발할 매수 5건을 엔진 기동 전에 선주입(버퍼 1024)한 뒤 기동,
  `ExecutionCh`에서 나오는 첫 5개 이벤트가 전부 취소인지 단언
  (`TestEngineProcessesCancelsBeforeNewOrders`). 수정 전 코드로 `-count=5` 실행 시
  5/5 전부 FAIL(무작위 select라 취소/체결이 섞임), 수정 후 `-count=5` 전부 PASS
  (결정론적) — "동등 select였다면 섞였을 것"이 실제로 증명됨.
- `go build`/`go vet`/전체 스위트(SKIP 0)/`-race`(matching·service·cmd) 전부 그린.
  **기존 엔진 단위·동시성·샤딩·정산·부트스트랩 통합 테스트 무수정 그린** — 처리
  로직·shutdown 도미노가 불변이라는 증거.
- 리뷰(opus, 매칭 엔진 코어 루프 리스크 고려): 변경 범위가 `Start()`의 select
  본문에만 국한됐는지, busy-loop이 없는지, 2단계 select의 `CancelCh` case가
  check-then-block 레이스를 정확히 닫는지, RED 증거(5/5 FAIL)가 실제 무작위
  select의 통계적 특성과 일치하는지(조작된/공허한 테스트가 아님을) 모두 diff에서
  직접 추적해 확인. Critical/Important 0건.

**이 조각은 취소 실패율/처리량 수치를 주장하지 않는다** — 급등 스파이크에서 취소
실패율이 실제로 낮아지는지의 실증은 ④(입장 정책 정교화)와 함께 ⑤(23번 벤치마크)에서
측정한다. ③의 성공 기준은 "취소가 신규 주문보다 먼저 처리됨을 결정론적으로 증명 +
회귀 그린"까지이며, 이 문서가 그것을 보인다. **P2(엔진 `ExecutionCh` 정지)는 이
조각의 범위 밖 — ①(주문 게이트)·②(DB 천장 상향)·④(목표 초과 시 빠른 거절)가
맡는다.**
