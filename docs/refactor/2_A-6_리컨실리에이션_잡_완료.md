## 왜 문제였나

"정합성 100%"가 **주장일 뿐 증명이 아니었다.** 원장(`ledger_entries`)과
지갑(`wallets`)이 실제로 일치하는지, 자산 총량이 보존되는지를 주기적으로 검사하는
장치가 없어서 — A-1/A-2처럼 아직 발견 못 한 정합성 버그가 조용히 잔고를 틀어지게
만들어도 아무도 모르는 상태였다. 모든 지갑 변동이 원장을 남기는 구조라 검증에
필요한 데이터는 이미 있었고, 검증 로직만 없었다.

## 어떻게 해결했나

`ReconciliationWorker`가 기동 직후 1회 + 1시간마다(환경변수로 조정 가능) 3가지를
검사하고, 위반을 `reconciliation_violations` 테이블(내구 기록) + Prometheus
게이지로 보고한다. **자동 교정은 하지 않는다** — decimal은 정확한 값이므로 tolerance
0, 불일치는 전부 조사 대상.

- **검사 1 (ledger_wallet)**: 지갑별 원장 델타 합 == `available/locked_balance`.
  단일 SQL(`LEFT JOIN LATERAL`, keyset 페이지네이션 500건)로 스냅샷 일관성을
  확보해 동시 정산에 의한 위양성을 차단. 레거시 이중 필드(`krw`/`quantity`) 폴백이
  만드는 알려진 괴리는, 가장 이른 원장 항목의 `after - delta`(추적 안 된 초기 잔액)가
  gap을 정확히 설명하고 locked gap이 0일 때만 `legacy_mismatch`로 분리 분류.
- **검사 2 (asset_conservation)**: 자산별 `Σ지갑잔액 + (KRW만) 누적수수료 ==
  Σ DEV_FUND 입금`. DEV_FUND가 코인도 충전하므로 심볼별 GROUP BY로 일반화 —
  전 자산의 총량 보존을 한 쿼리로 검증.
- **검사 3 (stale_market_order)**: 5분 넘게 PENDING/PARTIAL인 시장가 주문 탐지.
  시장가는 오더북에 rest하지 않으므로 존재 자체가 완료 유실의 증거 — A-1이 고친
  버그 클래스의 최종 안전망(재시도 소진 케이스 포함).
- 검사 쿼리 실패 시 게이지를 0으로 덮지 않고(가짜 정상 방지)
  `reconciliation_check_errors_total`만 증가.

설계 과정에서 리뷰로 잡은 함정: 원장 델타가 "유효 잔액" 기준이라 레거시 지갑이
영구 괴리를 만드는 문제(→ implied-initial 분류), 최초 분류 휴리스틱이 정반대로
동작하는 결함, 배치마다 원장 전체를 재집계하는 서브쿼리(→ LATERAL), 공유 테스트
DB에서 전역 0건 단언이 flaky해지는 문제(→ subject 필터 + 유니크 심볼 격리).

## 결과

- **위반 주입 → 탐지 체인 실증**: 진짜 원장 괴리 / 레거시 패턴 / 10분 경과 시장가
  3종을 주입하고 RunOnce 실행 → 각각 `ledger_wallet` / `legacy_mismatch` /
  `stale_market_order`로 정확히 분류·기록, 정상 지갑은 미기록, 위반 해소 후엔
  새 행이 쌓이지 않음을 실제 Postgres에서 확인.
- 분류 로직 단위 테스트 7건, 오케스트레이션 4건, 리포지토리 통합 7건, E2E 2건 —
  전체 스위트 통과. **직전 커밋에서 실패하던 CI도 이 작업의 픽스처 수정
  (시장가 매수는 `amount=0 AND quote_amount>0`, `ck_orders_shape_by_type`)으로
  그린 복구.**
