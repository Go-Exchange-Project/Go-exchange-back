# 7. B-1c 체결 브로드캐스트 배치화 — 완료

- **완료일**: 2026-07-16
- **상세 설계**: [2026-07-16-trade-broadcast-batching-design.md](../superpowers/specs/2026-07-16-trade-broadcast-batching-design.md)
- **전제**: B-4 정산 그룹커밋(6번) — 묶음의 원천.

## 왜 문제였나

[19번 벤치마크](../benchmarks/19-2026-07-16-b1-b4-batch-remeasurement.md) 해석 2:
B-1a(스냅샷 코얼레싱)는 심볼별 스냅샷 생성을 100ms당 1회로 캡핑했지만, **체결
(trade) 브로드캐스트는 캡핑 대상이 아니었다** — 정산 워커가 체결마다 개별로
발행했다. B-4 그룹커밋으로 처리량이 5배 이상 오르자 체결 수 자체가 급증했고,
legacy full-feed로 나가는 메시지 총량도 그만큼 늘어(측정 2 2,986만 건 →
측정 4 5,038만 건, +69%) WS fanout이 새로운 상대적 병목으로 부상했다(WS
300클라이언트에서 처리량 하락폭이 A −23.9% vs B −33.5%로 오히려 커짐).

## 어떻게 해결했나

**B-4 배치 = 브로드캐스트 배치.** 새 타이머·고루틴·버퍼 없이, 정산 워커의
손에 이미 들려 있는 "방금 함께 커밋된 체결들"을 그대로 브로드캐스트 단위로
재사용한다:

- `cmd/main.go`의 `broadcastSettledTrade`(단건)를
  `broadcastSettledTrades(trades []*model.Trade, ...)`(배치)로 교체. 워커
  큐가 심볼 해시 파티션이라 한 배치에 여러 심볼이 섞일 수 있으므로, **심볼별
  그룹핑(등장 순서·배치 내 순서 보존)** 후 심볼당
  `{"type":"trades","data":[...]}` 메시지 1건을 발행한다.
- 배열 원소는 기존 `"type":"trade"`의 `data` 객체와 필드 구성이 **정확히
  동일**(`coin_symbol`, `engine_sequence`, `engine_event_id`,
  `idempotency_key`, `price`, `quantity`, `fee_rate`, `buyer_fee`,
  `buyer_fee_asset`, `seller_fee`, `seller_fee_asset`, `time`) — 프론트
  `toTradeHistoryEntry`가 원소 단위로 그대로 재사용된다.
- `"type":"trade"` 단건 형식은 **완전히 은퇴**시켰다. 폴백(`settleTradeBatchWithFallback`
  실패 시 단건 재처리)·리플레이·단건 정산 경로(`processTradeSettlement`)
  모두 원소 1개짜리 `trades` 배열을 쓴다 — 와이어 형식이 하나만 남는다.
- **`internal/ws`(hub) 패키지는 무변경** — 심볼 라우팅(B-1b)·클라이언트
  관리 로직 그대로. 변경 지점은 `cmd/main.go`의 마샬 지점과 프론트엔드
  핸들러뿐이다.
- **배포 순서로 호환성 확보**: 프론트([Go-exchange-front](https://github.com/Go-Exchange-Project/Go-exchange-front)
  `49b9f74`)를 먼저 배포해 `trades` 배열 핸들러를 추가(기존 `trade` 단건
  처리는 유지 — 양형식 수용)한 뒤, 백엔드(`83d3f4c`)를 전환했다. 이 순서면
  어느 시점에도 체결 피드가 끊기지 않는다.

## 결과

**테스트**: `cmd/main_test.go`에 신규 3건 —
`TestBroadcastSettledTradesGroupsBySymbolPreservingOrder`(혼합 심볼 배치 →
심볼당 1메시지, 배치 내 순서 보존),
`TestBroadcastSettledTradesPayloadFormat`(형식 `trades`+배열, 원소 필드
기존과 동일), `TestBroadcastSettledTradesEmptyIsNoop`(빈 슬라이스 → 발행
0건). 기존 테스트 2건을 새 동작에 맞게 갱신
(`TestProcessTradeSettlementBroadcastsAppliedTrade`의 타입 어서션,
`TestSettleTradeBatchWithFallbackBroadcastsOnlyAppliedOnSuccess`의 브로드캐스트
횟수 — 같은 심볼의 Applied 결과가 배치 메시지 1건으로 묶이므로 2→1).

**회귀**: `go build ./...` 통과. `go test ./cmd/... ./internal/ws/... -count=1`
통과 — `internal/ws` 스위트가 무변경으로 그대로 통과해 hub 패키지 무영향을
증명한다. 통합 포함 전체 스위트 `go test ./... -count=1 -v`:
**390 PASS / 0 SKIP / 0 FAIL**.

**로컬 E2E**: 로컬 백엔드(임시 postgres) + 프론트 dev 서버를 기동하고,
`ws://localhost:8080/ws`에 직접 연결하는 스크립트로 실제 발행 메시지를
캡처했다(헤드리스 브라우저가 없어 프론트 UI 클릭 대신 이 방식으로 확인).
유저 2명 생성·충전 후 지정가 매도/매수 쌍 2회를 제출 → 실서버가
`"type":"trades","data":[...]`만 발행(레거시 `"type":"trade"` 0건)하고,
필드 구성이 기존과 동일함을 확인했다.
