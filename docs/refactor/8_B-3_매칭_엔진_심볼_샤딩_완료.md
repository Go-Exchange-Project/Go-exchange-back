# 8. B-3 매칭 엔진 심볼 샤딩 — 완료

- **완료일**: 2026-07-17
- **상세 설계**: [B-3 심볼 샤딩 설계](../superpowers/specs/2026-07-17-engine-symbol-sharding-design.md)
- **커밋**: `955f5ee`(Engine 인터페이스 분리) → `632455a`(ShardedEngine 코어) → `f3d2818`(파이프라인 전환)

## 왜 문제였나

[20번 벤치마크](../benchmarks/20-2026-07-16-multi-symbol-remeasurement.md)의 엔진
병목 판정: 매칭 연산 자체는 CPU의 0.52%로 저렴한데, **`order` 채널이 hold 8분
내내 예외 없이 버퍼 상한(1024)에 고정**돼 있었다. 단일 goroutine이 8개 심볼을
전부 직렬 처리하는 구조 자체가 처리량 캡(1,479 iter/s에서 포화)이었다 — CPU
프로파일만 봤다면 "엔진은 한가하다"로 오판했을, 직렬화로 인한 처리량 캡이다.

## 어떻게 해결했나

**기존 `MatchingEngine`은 무수정**(허용된 추가는 `SubmitOrder` 메서드 한 줄과
`Engine` 인터페이스 선언뿐). 새 `ShardedEngine`이 엔진 N개를 소유하고 라우팅한다:

- **first-seen 라운드로빈 배정** (`sync.Map` + `LoadOrStore`): 처음 보는 심볼을
  다음 샤드에 고정. 해시 대신 라운드로빈인 이유는 20번에서 해시 파티션의
  불균형(정산 워커 10개 중 2개에만 적체)을 실측했기 때문 — 소규모 심볼 수에서
  해시 충돌이 측정과 효과를 왜곡한다.
- **팬인 포워더**: 샤드별 Execution/Snapshot 채널을 샤드당 goroutine 1개가 머지
  채널로 복사 — 심볼 내 이벤트 순서 보존(한 심볼=한 샤드=한 포워더). 머지
  ExecutionCh는 블로킹 송신이라 A-3 백프레셔(outbox가 밀리면 매칭 블록)가 그대로
  성립하고, 머지 채널 close는 전 포워더 종료 후(WaitGroup) 1회 — graceful
  shutdown 도미노의 스코프만 확장됐다.
- **멱등성 안전**: `EngineEventID`가 인스턴스별 고유 `engineID`를 포함하므로 샤드
  N개가 각자 생성해도 충돌 없음(설계 시 사전 검증한 핵심 리스크 — 기존 코드가
  이미 안전했다).
- 샤드 수는 `GOEXCHANGE_ENGINE_SHARDS`(기본 `runtime.NumCPU()`). 기존 채널
  게이지 4개는 샤드 합산으로 대시보드 호환을 유지하고, 신규
  `matching_engine_shard_order_channel_length{shard}` 게이지로 샤드별 적체를
  관측한다(20번에서 병목을 잡아낸 관측 수단의 계승).

## 결과

- ShardedEngine 단위 7건(배정 안정성·라운드로빈 균등·심볼 내 순서 보존·샤드 간
  독립 매칭·취소 라우팅·스냅샷 캐시·Stop 드레인 유실 0) + 다심볼 동시 제출
  동시성 테스트, matching `-race` 통과.
- 전체 스위트 12개 패키지 그린(통합 포함, SKIP 0) + service·cmd `-race` 통과 —
  부트스트랩·outbox 통합 테스트가 무수정으로 통과해 종료 도미노와 write-ahead
  불변식이 유지됨을 확인. 기존 엔진 테스트도 무수정 통과(엔진 무수정의 증거).
- **처리량 효과는 이 문서에서 주장하지 않는다** — 실증은 21번 측정 사이클
  (A=전환 전, B=`GOEXCHANGE_ENGINE_SHARDS=8`, 다중 심볼 hold same-session A/B).
  판정 신호: ① 샤드별 order 채널이 1024 고정에서 풀리는지 ② 1,479 iter/s 캡
  돌파 여부 ③ 다음 병목 식별(pprof상 주문 생성 경로 DB 왕복 50%가 후보).
