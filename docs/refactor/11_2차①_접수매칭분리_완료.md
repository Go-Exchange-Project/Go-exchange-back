# 11. 2차 리팩토링 ① 접수-매칭 분리 — 완료

- **완료일**: 2026-07-19
- **설계 문서**: [① 접수-매칭 분리 설계](../superpowers/specs/2026-07-18-acceptance-matching-decoupling-design.md)
- **구현 계획**: [① 접수-매칭 분리 구현 계획](../superpowers/plans/2026-07-18-acceptance-matching-decoupling.md)
- **근거**: [22번 벤치마크](../benchmarks/22-2026-07-18-outbox-ab-and-spike.md) — 스파이크 시 `POST /orders` p95 14.8초.

## 왜 문제였나

`OrderService.CreateOrder`의 순서는 ① 검증 → ② DB 트랜잭션(영속화 + 자금 홀드,
커밋) → ③ `MatchingEngine.SubmitOrder(...)`(`OrderCh <-` **블로킹 송신**) → ④
200 반환이었다. 문제는 ③이다 — DB CPU가 포화되면 엔진 goroutine이 자신의
`ExecutionCh` 송신에서 멈추고, 그러면 `OrderCh`를 못 비워 블로킹 송신이 몇 초씩
쌓인다. 22번 벤치마크가 이를 정확히 실측했다(hold 구간 p95 14.86초, DB CPU
포화 시). **DB 병목이 HTTP 응답까지 역류하는 것이다.**

핵심 관찰(설계 문서): ②의 커밋 시점에 주문은 이미 **내구·정합적**이다 — 자금은
잠겼고(overspend 불가), 주문 행은 영속화됐다(PENDING). 크래시해도 부트스트랩이
PENDING/PARTIAL 주문을 재투입한다. 따라서 **응답은 ③(엔진 핸드오프)을 기다릴
이유가 없다** — 그건 저지연 매칭을 위한 인메모리 최적화일 뿐, 내구성 요건이
아니다.

## 어떻게 해결했나

접수 시퀀스를 4단계로 재정의했다(Task 1-4, 태스크 단위 TDD):

1. **Task 1 (`ea7e8e6`)**: `matching.Engine` 인터페이스에
   `TrySubmitOrder(order, within) bool`(바운디드 — `select`+타이머, `false`면
   `select`의 상호배타적 분기 특성상 주문이 채널에 **절대 들어가지 않았음**이
   보장됨)과 `IsIntakeAdmissible(coinSymbol) bool`(`OrderCh` 점유율 90%
   high-watermark 게이트)을 추가. `MatchingEngine`에 직접 구현, `ShardedEngine`은
   기존 `shardFor` 라우팅에 위임. **기존 블로킹 `SubmitOrder`는 무변경** —
   부트스트랩·리플레이 전용으로 계속 사용된다(그 경로엔 HTTP가 기다리지 않으므로
   블로킹이 옳다).
2. **Task 2 (`efb0552`)**: 신규 터미널 상태
   `model.OrderStatusRejected`("REJECTED") — 시스템 부하로 인한 접수 거절을
   유저 취소(`CANCELLED`)와 구분. **마이그레이션 불필요**: `orders.status`에
   기존 DB CHECK 제약이 없음을 확인했다(migration 001에는 amount/price CHECK만
   있고 status CHECK 부재 — enum 상수만 추가). `service.ErrorKindUnavailable` +
   `NewUnavailableErrorf` → `serviceErrorStatus`/`CodeForStatus`에 503 매핑 추가.
   방어 확인: `isCancellableOrderStatus`(PENDING/PARTIAL만 허용)와
   `validateOrderStatusForSettlement`(명시 케이스 외 `default`로 거부)가 코드
   변경 없이 REJECTED를 이미 올바르게 배제함을 확인.
3. **Task 3 (`b7cd3fc`)**: `config.OrderAcceptanceTimeoutFromEnv()`
   (기본 100ms, `GOEXCHANGE_ACCEPTANCE_TIMEOUT_MS`) — 기존 `OutboxBatchSizeFromEnv`
   패턴 그대로.
4. **Task 4 (`8d33424`, 핵심)**: `CreateOrder` 재작성.
   - **입장 게이트**(DB 작업 **전**): `IsIntakeAdmissible`이 false면 DB 무접촉
     즉시 503 — 지속 포화 시 DB 낭비 없는 빠른 거절.
   - **DB 트랜잭션**: 영속화 + 자금 홀드. **완전히 무변경**(`holdOrderAssets`
     그대로) — 자금 홀드는 여전히 동기·원자적(overspend 불가).
   - **바운디드 핸드오프**: `TrySubmitOrder(order, acceptanceTimeout)`로 블로킹
     `SubmitOrder`를 대체.
   - **보상**(게이트 통과 후 핸드오프가 레이스로 타임아웃하는 rare-rare 케이스):
     신규 `rejectAcceptedOrder`/`releaseInitialHold`가 `holdOrderAssets`의
     정확한 역을 수행(매수 지정가=`quoteAmountWithTradingFee(Price×Amount)`,
     매수 시장가=`order.QuoteAmount`, 매도=`order.Amount` — 전부 홀드 계산식과
     동일 표현식 재사용, `applyBuyOrderHold`가 아닌 `releaseBuyOrderHold` 등
     해제 전용 함수로) 홀드 전액 해제 + `REJECTED` 확정, 한 트랜잭션. 이 보상이
     동기로 안전한 이유: `TrySubmitOrder`의 `false`는 주문이 엔진에 **절대
     들어간 적 없음**을 보장하므로(Task 1의 불변식) 어떤 trade도 이 주문을
     참조할 수 없다 — 레이스가 원천적으로 불가능하다.
   - `cmd/main.go`에 `orderService.AcceptanceTimeout = config.OrderAcceptanceTimeoutFromEnv()`
     배선.

## 결과

- **재현 테스트 3종**(Task 4, 실 Postgres·모킹 없음): ① 정상(여유 시 200·PENDING·
  엔진 접수) ② 게이트 거절(포화 시 DB 무접촉 503 — 주문 0건·원장 0건까지
  단언) ③ 바운디드 타임아웃 보상(영속화·홀드된 뒤 핸드오프 타임아웃 →
  잔고 원복·`REJECTED`·원장에 `OrderHold`+`OrderRelease` 쌍 존재까지 단언,
  "최종 잔고가 우연히 맞다"가 아니라 홀드→해제 왕복 자체를 증명).
- `go build`/`go vet`/전체 스위트(통합 포함 SKIP 0)/`-race`(matching·service·cmd)
  전부 그린. **기존 `CreateOrder`·정산·부트스트랩 통합 테스트 무수정 그린** —
  블로킹 `SubmitOrder` 경로(부트스트랩·리플레이)가 전혀 영향받지 않았다는 증거.
- 리컨실리에이션: 보상 시나리오 테스트에서 `legacy_mismatch`(이 리포 테스트의
  직접 지갑 시딩 패턴이 만드는 기존·의도된 분류, 실제 결함 아님)를 제외한
  `ledger_wallet`(진짜 결함) 위반 0을 확인 — 이 좁힌 단언이 공허하지 않음을
  리뷰어가 `classifyLedgerWalletRow` 코드로 독립 검증했다.
- 자금 안전성: 홀드→보상 해제 계산식이 `holdOrderAssets`와 정확히 대칭임을
  리뷰어가 코드를 나란히 놓고 직접 대조해 확인(모든 매수/매도·지정가/시장가
  조합).

**이 조각은 처리량/지연시간 수치를 주장하지 않는다** — 접수 지연의 최종 실증은
②(자금 홀드 배칭)·③(취소 우선 경로)·④(입장 정책 정교화)와 함께 ⑤(23번
벤치마크)에서 측정한다. ①의 성공 기준은 "블로킹 제거 + 보상 정합 + 회귀
그린"까지이며, 이 문서가 그것을 보인다.

## API 계약 영향 (프론트 — 최소)

- **기존 계약 그대로**: `POST /orders`는 원래도 엔진 핸드오프(매칭 전)에서
  200을 반환했다 — 주문은 PENDING으로 접수되고 매칭은 비동기다.
- **신규**: 포화 시 **503**(`SERVICE_UNAVAILABLE`, 재시도 안내) 응답이 추가된다.
  목표 부하 이하에선 거의 발생하지 않고, 목표 초과의 안전 마진이다. 503 본문
  형식·`Retry-After` 등 세부 정책은 ④(입장 정책 정교화)의 몫.
