# 리팩토링 로드맵

2026-07-10 아키텍처 전수 리뷰(Fable5)에서 나온 개선 항목들의 진행 현황판입니다.
목표: **더 많은 사람이, 더 빠르게, 100% 정확도로 거래**할 수 있는 거래소.

## 운영 규칙

- 항목을 완료하면 아래 표의 상태를 ✅로 바꾸고, `N_항목이름_완료.md` 문서를
  이 디렉토리에 추가한다(N은 작업 완료 순서).
- 완료 문서에는 **왜 문제였는지 / 어떻게 해결했는지 / 결과가 어땠는지**를 적는다.
  상세 설계·검증 기록은 `docs/superpowers/specs/`의 해당 설계 문서에 있으므로
  완료 문서는 요약 + 링크로 유지한다.
- 우선순위 원칙: **정합성 구멍 → 정합성의 증명/보험 → 내구성 → 확장성** 순.
  병목이 아닌 곳을 먼저 최적화하지 않는다(16번 벤치마크의 교훈).

## 2차 리팩토링 — 가용성 100% (2026-07-18 착수)

1차(1~10번)로 **정합성**은 실전 급등락에서 위반 0으로 실증됐다([22번 2부](../benchmarks/22-2026-07-18-outbox-ab-and-spike.md)).
그런데 같은 22번이 **가용성 갭**을 드러냈다: 스파이크 시 취소 43.9% 실패(CancelCh
타임아웃)와 주문 접수 지연 초 단위(엔진 정지의 HTTP 역류). 둘 다 정합성이 아니라
**가용성** 문제다 — 돈이 오가는 서비스에서 가용성도 정합성과 같은 급으로 다뤄야 한다.

**목표(정직한 정의):** 동시 **~5,000 제출자(≈3,000건/s)** 까지 **① 즉시 접수(한 자릿수
ms p95) ② 취소 무실패 ③ 정합성 위반 0**을 급등 스파이크 주입으로 실증한다. 목표 초과
부하는 **매달림 없이 빠른 거절(503+재시도)** — "완료 또는 빠른 명확한 거절, 몇 초 매달림
0"이 하드 보장이다. 고정 하드웨어(스케일업 없음) 유지. 배칭으로 천장이 목표까지 닿는지
측정하고, 벽이면 인메모리 잔고로 승격(측정 기반 에스컬레이션).

> 근본 원인은 하나다: **DB CPU 포화 → ExecutionCh 만석 → 엔진 정지 → OrderCh/CancelCh
> 역류.** 취소 500·주문 14초·처리량 정체가 전부 이 한 뿌리의 세 증상이다.

## 리뷰에서 나왔지만 채택하지 않은 것

- **MSA 전환**: 매칭 엔진의 본질이 심볼당 단일 스레드 직렬화라 분산해도 병렬화되지
  않음. 모듈러 모놀리스 유지. 유일한 분리 후보는 WS market-data fanout(B-1 이후 재판단).
- **Kafka**: 단일 브로커는 Postgres COMMIT 대비 추가 내구성이 0인데 dual-write 문제를
  새로 만듦. A-3의 transactional outbox가 정석 관문(추후 CDC로 전환 경로 열림).
- **Redis**: 단일 프로세스에선 `atomic.Value`(인프로세스)가 항상 우월. WS를 다중
  인스턴스로 분리하는 시점에 pub/sub으로 첫 도입.
