# 10. A-4 취소-체결 레이스 + 시장가 매수 반올림 초과 소진 — 완료

- **완료일**: 2026-07-18
- **설계 문서**: [정합성 버그 2건 수정 설계](../superpowers/specs/2026-07-18-cancel-fill-race-and-market-buy-overspend-fixes-design.md)
- **구현 계획**: [정합성 버그 2건 수정 구현 계획](../superpowers/plans/2026-07-18-cancel-fill-race-and-market-buy-overspend-fixes.md)
- **발견 문서**: [bug-findings 2026-07-17](bug-findings-2026-07-17-cancel-fill-race-and-market-buy-overspend.md)

## 왜 문제였나

22번 벤치마크 사전 작업으로 만든 `loadtest/order-spike-single-symbol.js`의 로컬
스모크(유저 20명, 55초)에서 이 프로젝트가 처음으로 취소·시장가 주문을 부하
상태에서 실행하며 사전에 없던 정합성 버그 2건을 실증했다(발견 문서 참고):

1. **시장가 매수 초과 소진**: MARKET BUY가 여러 지정가 레벨에 분할 체결되면
   `matchMarketBuy`의 decimal `Div` 반올림이 예산을 epsilon(+1.8775e-9 관측)
   초과하는 체결을 만들었고, 그 실패를 기록하려는 `failed_market_completions`
   INSERT 자체가 CHECK 제약(`remaining_quote_amount >= 0`) 위반으로 또 실패했다
   — 실패의 안전망이 무음으로 붕괴. 로컬 스모크에서 서로 다른 주문 97건 재현.
2. **취소-체결 레이스(백로그 A-4)**: `OrderService.CancelOrder`가 DB 트랜잭션에서
   즉시 `CANCELLED`를 커밋한 뒤에야 매칭 엔진에 취소를 요청했다. 엔진이 이미
   체결시켰지만 outbox 파이프라인 지연으로 DB에 반영 전인 경우, 뒤늦게 도착한
   trade의 정산이 "status CANCELLED cannot be settled"로 실패(`failed_settlements`)
   하거나 엔진이 "이미 없음"을 반환해 취소 API가 500을 반환했다. 로컬 스모크에서
   취소 154건 중 500 21건(13.6%), `failed_settlements` 24건 재현.

## 어떻게 해결했나

### 시장가 매수 초과 소진 — 3계층 수정

1. **근본 (Task 2A, `a26c235`)**: `matchMarketBuy`의
   클램프 체결에서 `Div` 대신 `QuoRem`으로 quantity 자체를 예산 내로 내림(floor).
   `executionQuote`만 클램프하지 않는다 — 정산이 `price × quantity`로 재계산하므로
   quantity가 예산 내여야 일관된다. 클램프 후 `order.QuoteAmount ≥ 0`이 항상
   성립함을 결정론적 재현 테스트로 증명(수정 전 FAIL → 후 PASS). `matchMarketSell`은
   이 문제가 없어(수량 기준 소진) 무수정.
2. **안전망 (Task 2B, `1e87fc1`)**:
   `FailedMarketCompletionService.RecordFailure`에서 `RemainingQuoteAmount`를
   저장 전 `decimal.Max(0, ...)`로 clamp — 근본 수정과 독립적으로 "실패를
   기록하려다 CHECK 위반으로 또 실패"하는 이중 실패를 원천 차단. 동종 결함
   점검 결과 `failed_settlements`(`FailedSettlementService.RecordFailure`)는
   호출 전 `price`/`quantity` > 0을 이미 검증하고 있어 같은 패턴이 없음을 확인
   (수정 불필요, 통합 테스트로 확인).
3. **E2E 재현 확인 (Task 2C)**: 실 백엔드 + 실 Postgres로 다중 레벨 오더북에
   MARKET BUY를 분할 체결시켜 `failed_market_completions` 0건 신규,
   리컨실리에이션 위반 0건, 관련 에러 로그 0건을 확인.

### 취소-체결 레이스 — 취소 시퀀서화 (사용자 확정: "정석")

**주문 상태 전이 소유권을 엔진 시퀀스로 옮겼다.** 취소를 체결과 같은 등급의
엔진 이벤트로 만들어, "이 주문의 선행 체결들 → 취소"가 엔진 FIFO로 자동
정렬되게 했다:

1. **Task 1A (`f94f151`)**: outbox 이벤트 타입
   `ORDER_CANCELLED` 추가 + CHECK 제약 마이그레이션(`004_order_cancelled_event.sql`,
   gorm AutoMigrate는 기존 CHECK를 갱신하지 않으므로 필수).
2. **Task 1B (`6c39199`)**: 엔진이 취소로 오더북에서
   실제 제거(`Removed: true`)했을 때만 `ExecutionCh`에 `OrderCancelled` 이벤트를
   방출(단일 스레드 순서상 선행 체결들 뒤에 자동 enqueue). `EngineEventID`는
   `Trade`와 동일하게 엔진 시퀀스에서 부여.
3. **Task 1C (`745b680`)**: `NewTradeOutboxEvent`/
   `ExecutionEventFromOutbox`에 `ORDER_CANCELLED` 인코딩/디코딩 분기 추가 —
   리플레이는 기존 라우팅을 그대로 타 자동 커버.
4. **Task 1D (`d9297d0`)**: 신규
   `OrderService.ProcessOrderCancellation` — 엔진이 방출한 `OrderCancelled`를
   정산 파이프라인에서 확정(잔여 = 최신 `Amount - FilledAmount` 계산 → hold
   해제 → `CANCELLED` 커밋, 멱등). `cmd/main.go`의 `processExecutionEvent`에
   분기를 추가하고, **`executionEventCoinSymbol`에도 분기를 추가**해 취소
   이벤트가 그 주문의 체결들과 같은 심볼 정산 워커로 라우팅되게 했다(이게
   빠지면 FIFO 순서 보장이 워커 분산으로 깨진다 — 가장 놓치기 쉬운 지점이었다).
5. **Task 1E (`33ea704`, 레이스 해소의 핵심)**:
   `CancelOrder`를 "검증 + 엔진 접수"로 재작성 — DB에서 hold 해제·CANCELLED
   커밋을 완전히 제거하고 `ProcessOrderCancellation` 단독 소유로 넘겼다.
   엔진의 실제 "이미 없음" 신호(`ErrCancelOrderNotFound`)를 409로 정확히
   매핑해 기존 500 버그를 제거. **실제 엔진 + 실제 outbox writer(Postgres
   커밋) + 실제 정산, 모킹 없이** trade가 outbox에 커밋됐지만 아직 정산 전인
   "in-flight 창"에 취소를 접수시켜, ① `failed_settlements` 0건 ② outbox ID
   순서로 trade가 cancel보다 먼저 처리됨 ③ 정산 성공을 증명하는 통합 테스트로
   레이스 해소를 확인(`-race` 5회 반복 안정).
6. **Task 1F (`e0be670`)**: 프로덕션 코드 무변경 —
   1A-1E의 write-ahead/크래시 내구성/셧다운 드레인 설계에 실제 갭이 없음을
   검증 테스트로 확인. 커밋 후 크래시(리플레이가 outbox ID 순서로 trade→cancel
   처리해 완결, 재리플레이 멱등)는 신규 테스트로, 커밋 전 크래시(부트스트랩
   정상 재투입)는 기존 커버리지로 충분함을 코드 추적으로 확인, 셧다운 드레인은
   `CancelCh` 경로의 커버리지 공백 1건을 신규 테스트로 메웠다.

## 결과

- 재현 테스트가 수정 전 FAIL → 후 PASS로 실증된 항목: 버그2 엔진 클램프(2A),
  안전망 clamp(2B), 버그1 레이스 해소(1E)의 핵심 통합 테스트 — 전부 실제
  Postgres/엔진/outbox를 구동한 통합 테스트이며 모킹 없음.
- `go build`/`go vet`/전체 스위트(통합 포함 SKIP 0)/`-race`(matching·service·cmd)
  전부 그린. 기존 계약 테스트(매칭 단위·동시성, outbox writer·리플레이·크래시,
  정산 통합, graceful shutdown 도미노) 무수정 그린.
- 각 태스크에서 리컨실리에이션(A-6) 위반 0을 통합 테스트로 확인.

### API 계약 변경 (프론트 영향 — 이 세션 범위 밖)

`DELETE /orders/:id`(취소)가 **즉시 확정 → 접수 기반**으로 바뀌었다:
- 응답 형식(필드)은 유지되지만 `status`/`released_amount` 등은 이제 "접수 시점
  기준 추정"이다. 취소 요청 직후 `GET /orders/:id`를 조회하면 파이프라인 지연만큼
  잠깐 `PENDING`/`PARTIAL`로 보일 수 있고, 곧 `CANCELLED`로 전환된다.
- `released_asset`/`released_amount`는 검증 시점 스냅샷 기준 추정치이며, 실제
  확정 해제량의 진실의 원천은 `ProcessOrderCancellation`이 만드는 원장
  (`LedgerEntryTypeOrderRelease`) 기록이다. 드물게 취소 접수와 거의 동시에
  선행 체결이 끼어들면 추정치가 실제보다 클 수 있다(코드 주석에 명시).
- 프론트가 "접수→확정" 폴링 UX를 반영해야 하는지는 별도 리포 프론트 작업으로
  남는다(이 세션 범위 밖).
