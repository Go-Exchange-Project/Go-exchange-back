# 1. A-1/A-2 정산 정합성 버그 수정 — 완료

- **완료일**: 2026-07-11
- **커밋**: `b35839f` (fix(settlement): 시장가 완료 레이스와 지갑 락 데드락 정합성 버그 수정)
- **상세 설계·검증**: [docs/superpowers/specs/2026-07-11-settlement-consistency-fixes-design.md](../superpowers/specs/2026-07-11-settlement-consistency-fixes-design.md)

## 왜 문제였나

아키텍처 리뷰에서 "정합성 100%" 목표에 정면으로 위배되는 **살아있는 버그 2건**이 발견됐다.

**A-1. 시장가 완료(MarketOrderDone) 레이스 → 고객 자금 영구 동결.**
정산 워커 10개가 단일 `ExecutionCh`를 경쟁 소비했다. 엔진은 시장가 체결 trade들을
emit한 뒤 마지막에 완료(Done) 이벤트를 emit하는데, 워커들이 병렬로 집어가므로
**trade 정산이 끝나기 전에 Done이 먼저 처리**될 수 있었다. 이때 `CompleteMarketOrder`가
conflict를 반환하면 로그만 남기고 버렸다 — Done 이벤트는 엔진 메모리에만 존재하므로
여기서 잃으면 복구 불가였고, 시장가는 취소가 불가능해서 **잔여 hold가 영구 locked**로
남았다. 부하가 높을수록(=급등락일수록) 확률이 올라가는 최악의 타이밍 버그.

**A-2. 지갑 락 순서 미정렬 → 데드락 → 체결 영구 미정산.**
`SettleTrade`가 지갑을 buyer KRW → buyer coin → seller KRW → seller coin 순서로
잠갔다. 두 유저가 서로 반대 역할인 두 체결이 동시에 정산되면 AB-BA 데드락이
성립했고, Postgres가 abort한 정산은 `failed_settlements`에 기록만 되고 **자동
재시도가 없어서** 일시적 오류인데도 체결이 영구 미정산으로 남았다 — 매수자는
코인을 못 받고 매도자는 KRW를 못 받은 채.

## 어떻게 해결했나

- **심볼 파티셔닝 정산 워커**: 워커마다 전용 채널을 배정하고, 디스패처가
  `FNV-1a(CoinSymbol) % N`으로 라우팅. 같은 심볼의 이벤트는 항상 같은 워커가
  FIFO로 처리해 엔진이 만든 순서(trade들 → Done)가 보존된다. 엔진 코드는 무수정.
- **완료 실패의 내구 기록**: 완료가 conflict/transient로 실패하면 in-place 백오프
  (50/100/200ms) 재시도 → 소진 시 신규 `failed_market_completions` 테이블에 기록 →
  재시도 워커가 처리. "로그만 남기고 유실" 경로 제거.
- **지갑 락 2단계 정렬**: 1차로 락 없이 4개 지갑 ID만 확보(없으면 생성), 2차로
  `WHERE id IN (...) ORDER BY id FOR UPDATE`로 항상 같은 순서로 일괄 잠금 —
  AB-BA 데드락 원천 차단. 잔고 산술은 2차에서 잠근 행으로만.
- **transient 자동 재시도**: SQLSTATE(40P01/40001/55P03) 기반 판정(에러 메시지
  문자열은 lc_messages에 따라 번역될 수 있어 코드로만 판정), 정산 경로 in-place
  재시도 + 10초 주기 `SettlementRetryWorker` 2차 방어선(5회 초과는 운영자 확인).
  멱등성 키가 원본 그대로 보존되므로 재시도해도 이중 정산이 구조적으로 불가능.

## 결과

- **데드락 회귀 테스트로 전후 실증**: 역할이 교차하는 두 정산을 30라운드 동시 실행 —
  수정 전 코드(HEAD 스왑)에서는 `deadlock detected (SQLSTATE 40P01)` 반복 재현,
  수정 후에는 **데드락 0건, 60건 전부 정산 성공**.
- 같은 심볼의 trade→Done 순서 보존, 완료 재시도/내구 기록, 재시도 워커 로직에
  대한 단위·통합 테스트 ~20건 추가, 전체 스위트(실제 Postgres 통합 포함) 통과.
- 부수 확보물: `failed_settlements`에 엔진 메타데이터(engine_sequence/event_id/
  traded_at) 보존, 워커별 큐 적체 게이지(`settlement_worker_queue_length`).
