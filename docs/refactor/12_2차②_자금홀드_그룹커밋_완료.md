- **완료일**: 2026-07-20
- **설계 문서**: [② 자금 홀드 그룹커밋 설계](../superpowers/specs/2026-07-19-order-hold-group-commit-design.md)
- **구현 계획**: [② 자금 홀드 그룹커밋 구현 계획](../superpowers/plans/2026-07-19-order-hold-group-commit.md)
- **근거**: [21번](../benchmarks/21-2026-07-17-b3-sharding-remeasurement.md)·[22번](../benchmarks/22-2026-07-18-outbox-ab-and-spike.md) — DB CPU가 병목(idle 4.5%), pprof에서 `holdOrderAssets`가 `CreateOrder` 경로의 26%.

## 왜 문제였나

①(접수-매칭 분리)은 응답을 엔진 매칭에서 떼어냈지만, 접수 자체는 여전히 동기 DB
트랜잭션(주문 영속화 + 자금 홀드)을 거친다. 이 홀드가 **주문마다 DB 왕복 여러 회**
(지갑 FOR UPDATE → 잔고 확인 → available→locked UPDATE → 원장 INSERT)를 쓰고,
21·22번에서 이 경로가 DB CPU를 포화시켜 처리량 천장을 만든다. 목표 용량(동시
~5,000 제출자 ≈ 3,000건/s)은 이 천장을 넘는다. B-4(정산 그룹커밋)가 정산 경로에서
DB 왕복을 배치로 N:1 상각해 효과를 실증했듯, **같은 기법을 주문 생성 홀드에
적용**해 천장을 목표 위로 올리는 것이 목표다.

B-4보다 까다로운 두 지점: (1) 정산 이벤트는 단일 워커 큐로 흘러 자연스러운 배칭
지점이 있었지만, 주문 홀드는 N개 동시 HTTP goroutine에서 각자 발생 — 모으는 계층이
새로 필요했다. (2) B-4의 trade 실패는 드물어 "배치 전체 롤백 → 단건 폴백"으로
충분했지만, 주문 홀드의 잔고 부족은 흔한 정상 케이스라 배치 안에서 개별 골라내야
했다.

## 어떻게 해결했나

새 `HoldCoordinator`(단일 goroutine)가 HTTP 홀드 요청을 ~5ms 창으로 모아 한
트랜잭션(`HoldBatch`)에 persist+hold하고 요청별 결과를 시그널한다(Task 1-5,
태스크 단위 TDD):

1. **Task 1 (`34ce448`)**: `OrderRepository.CreateOrders([]*model.Order) error` —
   GORM 슬라이스 create로 배치 INSERT, 각 원소 `.ID`가 RETURNING으로 채워짐.
2. **Task 2 (`1ce2b28`)**: 기존 `CreateOrder`의 단건 persist+hold 트랜잭션 블록을
   `persistAndHold(...)` 패키지 함수로 추출(동작 완전 불변) — no-coordinator 경로와
   배치 실패 폴백이 공유하는 단건 경로, 정합성의 진실.
3. **Task 3 (`ba74ce8`, 정합성 코어)**: `HoldBatch(orders) ([]holdResult, error)`.
   지갑 ID 오름차순 `LockByIDs`(정산 `lockSettlementWallets`와 동일 순서 — 데드락
   불성립), 순차 fold-검증(통과분만 배치 INSERT/홀드, 부족분은 `ConflictError`로
   개별 격리·행 0), 원장 엔트리는 fold 전 계산(delta = update − 현재잔고) 후
   `foldWalletBalanceUpdate`로 다음 주문이 반영된 잔고를 보게 함, `ReferenceID`는
   배치 INSERT 후 실제 `order.ID`로 채움. 트랜잭션 실패 시 전체 `orders`의 `.ID`를
   0으로 리셋(B-4 `8b3007f`의 phantom ID 교훈 재적용).
4. **Task 4 (`ddc383f`)**: 코디네이터 동작 — `Run`/`collectBatch`(OutboxWriter
   패턴 미러링)/`processBatch`(배치 실패 시 `persistAndHold` 건별 폴백)/`Submit`(
   제출은 `select`+`default`로 바운디드, 만석 시 즉시 503; 결과 대기는 **무타임아웃**
   — 대기를 끊는데 코디네이터가 나중에 커밋하면 고아 주문이 생기므로)/`Shutdown`(
   입력 close → 잔여 drain → 반환). `config.HoldBatchSizeFromEnv`(기본 64),
   `hold_batch_size`/`hold_batch_fallbacks_total`/입력 길이 게이지 메트릭.
5. **Task 5 (`0a5dd0b`)**: `OrderService.HoldCoordinator` 필드 추가 — nil이면 기존
   `persistAndHold` 직접 호출(기존 테스트 전부 무수정 그린), 있으면 `Submit`으로
   위임하고 **반환된(ID 채워진) 포인터**를 이후 엔진 핸드오프·응답에 사용. `cmd/main.go`
   배선: 코디네이터 생성·기동·주입·게이지 등록, 그리고 **종료 순서**를
   `srv.Shutdown(httpCtx)`(동기, in-flight `CreateOrder` 드레인 완료까지 블로킹) →
   `holdCoordinator.Shutdown()` → `me.Stop()` 순으로 고정 — HTTP가 먼저 완전히
   멈추므로 `close(c.input)` 시점에 `Submit`이 진행 중일 수 없어 `select`의
   closed-channel-send panic 경로가 성립하지 않는다.

## 결과

- **등가성**(핵심 불변식): `HoldBatch`로 처리한 주문 시퀀스와 같은 시퀀스를
  `persistAndHold` N회로 순차 처리한 결과가 지갑 잔고·원장 델타/ReferenceID·주문
  상태까지 필드 단위로 일치함을 실 Postgres 통합 테스트로 증명
  (`TestIntegrationHoldBatchMatchesSequentialSingleHold`, 같은 유저 다중 주문
  fold 포함).
- **개별 격리**: 잔고 충분/부족 혼합 배치에서 통과분만 홀드·주문 행 생성, 부족분은
  `ConflictError`·행 0(`TestIntegrationHoldBatchIsolatesInsufficientFunds`,
  `TestIntegrationHoldBatchFoldsSameUserBalance`).
- **폴백**: 배치 트랜잭션 실패(DB 이상 등, 드묾) 시 각 요청이 개별
  `persistAndHold`로 재처리됨을 확인(`TestIntegrationHoldCoordinatorFallsBackOnBatchError`,
  DB 강제 종료로 재현).
- **바운디드 제출 + graceful shutdown**: 입력 만석 시 즉시 503
  (`TestHoldCoordinatorSubmitReturnsUnavailableWhenInputFull`), `Shutdown()` 호출 시
  이미 제출된 요청 전부가 유실 없이 시그널됨을 확인(`TestIntegrationHoldCoordinatorShutdownDrains`,
  20건 전량 검증).
- **CreateOrder 통합**: 코디네이터 경유 경로가 실제로 `Submit`/`HoldBatch`를 타는지
  (폴백이 아님) 정상 홀드·잔고부족 양쪽으로 확인, 리컨실리에이션(`legacy_mismatch`
  제외) 위반 0(`TestIntegrationCreateOrderViaHoldCoordinator`).
- `go build`/`go vet`/전체 스위트(SKIP 0, 통합 포함)/`-race`(matching·service·cmd)
  전부 그린. **기존 `CreateOrder`·정산·부트스트랩·outbox·①(접수-매칭 분리) 통합
  테스트 무수정 그린** — 코디네이터가 nil인 기존 테스트 구성에 전혀 영향을 주지
  않았다는 증거.
- 5개 태스크 모두 서브에이전트 리뷰 통과(Critical/Important 0건). Task 3(정합성
  코어)·Task 4(동시성)·Task 5(종료 순서 — 프로덕션 panic 리스크)는 fund-safety/
  concurrency 리스크를 고려해 opus 모델로 리뷰, 세 태스크 모두 브리프의 핵심 불변식을
  코드 라인 단위로 직접 추적해 확인했다(리포트 신뢰 없이 diff 자체를 검증).

**이 조각은 처리량/지연시간 수치를 주장하지 않는다** — 배칭이 실제로 발동해 천장이
오르는지의 실증은 ③(취소 우선 경로)·④(입장 정책 정교화)와 함께 ⑤(23번 벤치마크)에서
`hold_batch_size` 분포로 측정한다. ②의 성공 기준은 "등가성 + 격리 + 폴백 + 회귀
그린"까지이며, 이 문서가 그것을 보인다.

## API 계약 영향 (프론트 — 없음)

`POST /orders`의 요청/응답 계약은 변하지 않는다. 코디네이터는 서버 내부 구현
디테일이며, 성공/실패(잔고부족 409·포화 503)의 의미와 형식은 ①에서 이미 확립된
그대로다.

## 남은 것

없음 — ②는 Task 1-6 전부 완료. 다음은 ③(취소 우선 경로), ④(입장 정책
정교화 — ①의 엔진-유입 게이트와 ②의 코디네이터 입력 만석 두 입장 지점 통합·
Retry-After), ⑤(23번 실증) — 전부 이 세션 범위 밖. 단일 코디네이터가 병목으로
