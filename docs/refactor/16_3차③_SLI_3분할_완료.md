- **설계 문서**: [3차 ③ SLI 3분할 설계](../superpowers/specs/2026-07-24-sli-three-way-split-design.md)

## 왜 필요했나

23번 벤치마크에서 "가용성 ~100% / 업무 성공 ~7% / 취소 성공 ~51%"가 뒤섞여
"9,800 VU까지 버텼다"가 무슨 뜻인지 혼선이 났다. `_workspace/loadtest/order-spike-availability.js`
는 order/market/cancel/503 Counter는 있었지만, 주문 503이
`http.expectedStatuses(200,201,503)`으로 정상 취급돼 `http_req_failed`만으로는
① 서버가 제한 시간 안에 응답했는가 ② 주문이 실제로 접수됐는가 ③ 취소가 인프라
문제 없이 성공했는가를 구분할 수 없었다. 이 셋은 과부하에서 정반대로 갈린다 —
서버가 안 매달리고 빠르게 503을 내면 응답 가용성은 높지만 업무 성공률은 낮다.
그 차이가 곧 셰딩의 영향이다. 3차 ①(취소 500 제거)·②(처리량)를 고치기 전에,
그 효과를 정확히 잴 측정 토대를 먼저 깔아야 했다.

## 어떻게 해결했나

k6 `Rate` 세 개를 응답 처리 지점에서 인라인으로 `.add()`한다. 분류 로직은
순수 함수로 빼서(`sli-classify.js`) k6 셀프체크(`sli-classify.selftest.js`)가
판정표 전 행을 assert로 덮는다 — "느린 2xx를 가용으로 잘못 세는" 류의 조용한
버그를 막는다.

### 세 지표

1. **`sli_order_response_availability`** — `(주문 응답 ∈ {200,201,503}) AND
   res.timings.duration ≤ RESPONSE_SLO_MS`. 접수 여부와 별개로 서버가 제한
   시간 안에 계약된 응답을 반환했는지. **느린 2xx도 실패**(24초 뒤 받은 201은
   업무 성공엔 포함되나 응답 가용성엔 불포함).
2. **`sli_order_business_success`** — `주문 2xx(200/201)`. 실제로 접수된
   비율(= 1 − 셰딩률). 빠르게 반환된 503도 여기선 실패.
3. **`sli_cancel_success`** — `취소 200 / (취소 200 + 취소 인프라 실패)`.
   404/409(이미 체결/오더북에서 사라짐 — 정상 경쟁 결과)는 분모 제외.
   **500만이 아니라 status 0·5xx·예상 밖 상태 전부**를 인프라 실패로 포함
   (k6 타임아웃·연결 실패가 분모에서 빠지면 지표가 실제보다 좋아 보인다).

### 판정표

주문:

| 응답 | 응답 시간 | 응답 가용성 | 업무 성공률 |
|---|---|---|---|
| 200/201 | ≤ SLO | 성공 | 성공 |
| 200/201 | > SLO | **실패** | 성공 |
| 503 | ≤ SLO | 성공 | **실패** |
| 503 | > SLO | 실패 | 실패 |
| status 0/기타 | 무관 | 실패 | 실패 |

취소:

| 응답 | 처리 |
|---|---|
| 200 | 성공 추가 |
| 404/409 | 집계 제외 |
| status 0·5xx·기타 | 실패 추가 |

### `RESPONSE_SLO_MS` — 벤치마크 경계이지 프로덕션 SLO 선언이 아님

```js
const RESPONSE_SLO_MS = parseInt(__ENV.RESPONSE_SLO_MS || '1000', 10);
```

기본 **1000ms**. 근거 — 23번 실측: B 주문 2xx p95 hold ~130ms·버스트
~228ms, 503 대부분 sub-ms, A 2xx p95 ~24초. 1000ms는 B 버스트 p95의 ~4.4배
여유라 일시적 네트워크 흔들림을 과실패 처리하지 않으면서 초 단위 매달림은
명확히 불합격시킨다. **이 값은 "스파이크 벤치마크 응답 경계 기본값"으로
정의**했을 뿐, 지금 프로덕션 99.9% SLO로 선언하지 않는다 — 운영 트래픽 데이터
확보 후 별도로 승격할 사안이다.

### 타이밍 — Retry-After sleep은 분류 뒤에

응답 가용성 시간은 `res.timings.duration`(서버 응답 시간)만 쓴다. 503 이후
`Retry-After`를 존중하는 `sleep()`은 클라이언트 백오프이지 서버 지연이 아니므로
포함하지 않는다 — `submitOrder`에서 분류·`.add()`를 `res` 수신 직후, sleep 전에
수행해 자연히 분리했다.

### 변경 범위 (외과수술적)

- **Go 코드·서버 메트릭 무변경** — k6 스크립트와 문서만.
- 기존 `Counter`(`custom_order_success` 등)·`http.expectedStatuses`는 그대로
  유지. Rate는 추가일 뿐 대체가 아니다.
- **Rate threshold를 걸지 않는다** — 이 단계는 정의+기준선 수집. 합격 목표는
  재측정 후.
- **whole-run SLI로 시작** — 스테이지별 태깅 없음. 23번 볼륨상 hold+버스트가
  지배해 "스파이크 하 세 지표"를 그대로 보여준다.
- **서버측 SLI 메트릭·Grafana 대시보드는 만들지 않았다** — 업무·취소 성공률은
  기존 `HTTPRequestsTotal{path,method,status}`에서 PromQL로 유도 가능(아래).
  응답 가용성은 클라이언트 지연이 핵심이라 k6가 적합.

## PromQL 유도 예시 (서버측, 참고용)

업무·취소 성공률은 신규 서버 메트릭 없이 기존 `HTTPRequestsTotal{path,method,status}`
에서 유도할 수 있다(k6의 `sli_order_business_success`/`sli_cancel_success`와
동치, 관측 시점·모수만 다름 — k6는 클라이언트 측 시도 기준, 이 쿼리는 서버가
실제로 받아 응답한 요청 기준):

```promql
# 업무 성공률(주문 접수율) — 2xx / 전체
sum(rate(http_requests_total{path="/orders", method="POST", status=~"2.."}[1m]))
/
sum(rate(http_requests_total{path="/orders", method="POST"}[1m]))

# 취소 성공률 — 200 / (200 + 인프라 실패, 404·409 제외)
sum(rate(http_requests_total{path="/orders/:id", method="DELETE", status="200"}[1m]))
/
sum(rate(http_requests_total{path="/orders/:id", method="DELETE", status=~"200|5.."}[1m]))
```

(정확한 라벨명은 `internal/metrics/metrics.go`의 `HTTPRequestsTotal` 정의를
따른다 — 위는 유도 형태의 예시.) 응답 가용성(`sli_order_response_availability`)
은 클라이언트가 관측한 왕복 지연이 핵심이라 서버 메트릭만으로는 유도할 수 없다
— k6가 계속 이 지표의 원천이다.

## 결과

- **셀프체크 PASS**: `sli-classify.selftest.js`가 주문 판정표 6행 + 취소
  판정표 6행 전부를 assert로 검증, `k6 run` 종료 코드 0(`SLI selftest PASSED`).
- **로컬 스모크 확인**: 로컬 backend(`go run ./cmd` + 테스트 Postgres) 기동 후
  소규모(`VU 5`) 오버라이드로 실행 — 요약에 세 Rate가 정상 출력됐고 값이 sane
  했다(저부하라 셋 다 ~100%). **`sli_cancel_success`(19/19)가
  `cancel_already_filled`(7건, 404/409)를 분모에서 자연 제외**함을 실제
  런에서 확인 — 순수 함수 셀프체크만이 아니라 실제 배선까지 검증됐다.
- Go 코드 무변경이므로 CI(Go 빌드·테스트·vet)는 회귀 없이 그대로 통과.
