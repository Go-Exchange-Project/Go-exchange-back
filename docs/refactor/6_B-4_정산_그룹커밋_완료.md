# 6. B-4 정산 그룹커밋 — 완료

- **완료일**: 2026-07-16
- **상세 설계**: [docs/superpowers/specs/2026-07-15-settlement-group-commit-design.md](../superpowers/specs/2026-07-15-settlement-group-commit-design.md)
- **전제**: 심볼 파티셔닝 정산 워커(A-3) — 같은 심볼은 항상 같은 워커가 FIFO 처리.

## 왜 문제였나

trade 1건 = 정산 트랜잭션 1개였고, 트랜잭션 하나가 DB 왕복 약 12회(멱등성 키
조회, trade INSERT, 주문 락 2회, 지갑 ID 확보 4회, 지갑 락, 주문 UPDATE 2회,
지갑 BatchUpdateBalances, 원장 CreateMany, outbox 마킹) + COMMIT 1회를 썼다.
서버-DB가 별도 VM(VPC)이라 왕복이 지배 비용인 것은 추측이 아니라 실측이다:
[18번 벤치마크](../benchmarks/18-2026-07-13-outbox-markprocessed-absorption.md)에서
**왕복 단 1회(MarkProcessed)를 트랜잭션에 흡수한 것만으로 처리량 +20.6%**가
나왔다. trade N건을 한 트랜잭션으로 묶으면 왕복이 12×N → 배치당 상수 ~9회로
줄어든다는 것이 이번 작업의 동기다.

## 어떻게 해결했나

- **배치 수집** (`cmd/main.go`의 `collectTradeBatch`): 첫 이벤트는 블로킹
  수신, 이후 큐에 **이미 쌓인** trade만 논블로킹으로 최대 32건(`settlementBatchMaxSize`,
  하드코딩·타이머 없음) drain. `MarketOrderDone`을 만나면 배치를 끊고 pending으로
  돌려줘 엔진 방출 순서(trade들 → Done)를 보존한다.
- **`SettleTradeBatch`** (`internal/service/settlement_batch.go`): 한
  트랜잭션 안에서 8단계 — 중복 분리(idempotency_key IN) → 신규 trade 배치
  INSERT(ON CONFLICT DO NOTHING) → 배치 내 고유 주문 ID 오름차순 일괄 락 →
  참가자 지갑 키 일괄 확보(FindByKeys + 부족분 CreateZeroBalanceWallets) →
  지갑 ID 오름차순 일괄 락 → trade를 큐 순서대로 메모리에서 순차 산술 →
  주문/지갑 배치 쓰기 + 원장 CreateMany → outbox 일괄 마킹. 검증·산술은
  단건 경로의 기존 헬퍼(`applyTradeFill`, `settleBuyerKRW`,
  `creditBuyerCoinWithAcquisitionCost`, `settleSellerCoin`, `creditAvailable`,
  `ledgerEntryFromWalletUpdate` 등)를 복제 없이 그대로 재사용한다 — 로직이
  갈라지면 정합성 등가 불변식이 시간이 지나며 깨지기 때문이다.
- **실패 격리**: 배치 트랜잭션이 어떤 이유로든 실패하면 전체 롤백(자금
  무변동) 후 **같은 배치를 기존 단건 `SettleTrade` 경로로 1건씩 재처리**
  (`settleTradeBatchWithFallback`). 불량 trade 1건만 `FailedSettlement`로
  빠지고 나머지 N−1건은 정상 정산된다. 락 순서는 주문 먼저(ID 오름차순
  일괄) → 지갑 나중(ID 오름차순 일괄)으로 단건 경로와 동일 — 이것이
  데드락 불성립 논증의 근거다(상세: 설계 문서의 "데드락 분석").
- **단건 `SettleTrade`는 시그니처·동작 모두 불변** — 리플레이·재시도
  워커·폴백이 계속 사용한다.

## 결과

**테스트** (4개 태스크에 걸쳐 신규 추가, 전부 통과):

- Task 1 (리포지토리 배치 프리미티브): `OrderRepository.LockByIDs` /
  `BatchUpdateExecutions`, `WalletRepository.FindByKeys` /
  `CreateZeroBalanceWallets` — 통합 테스트 6건(행 순서·개수 불일치 에러·멱등
  생성 포함).
- Task 2 (`SettleTradeBatch`): 통합 테스트 5건 —
  `TestIntegrationSettleTradeBatchMatchesSequentialSingleSettlement`가
  **정합성 등가 불변식**(같은 trade 시퀀스를 배치/단건 양 경로로 정산했을 때
  지갑·원장·주문 최종 상태가 필드 단위로 동일)을 회귀 검증하고, 나머지가
  멱등성(배치 내 부분/전체 중복), 실패 시 전체 롤백, outbox 일괄 마킹을 검증.
- Task 3 (메트릭): `settlement_batch_size` 히스토그램·
  `settlement_batch_fallbacks_total` 카운터 단위 테스트 2건.
- Task 4 (`cmd/main.go` 워커 통합): `collectTradeBatch`(드레인·빈 큐·Done
  경계·상한 캡·채널 닫힘) + `settleTradeBatchWithFallback`(폴백·성공 시
  Applied만 브로드캐스트) 단위 테스트 7건 + 기존 워커 루프 테스트 조정.

**graceful shutdown 도미노** (Task 5, 코드 리뷰로 확인 — 새 테스트 아님):
엔진 `Stop()`(`cmd/main.go:343`) → `ExecutionCh` close → `OutboxWriter.Run()`이
잔여 배치 flush 후 반환(`internal/service/outbox_writer.go:49-61`) →
`cmd/main.go:204-208`에서 모든 `settlementQueues` close → 각 워커의
`collectTradeBatch`가 닫힌 채널을 만나면 `open=false`를 반환하되 **이미 모은
배치는 그대로 반환**(`cmd/main.go:501-504`) → 워커가 그 배치를 정산한 뒤
`return`(`cmd/main.go:184-187`) → `settlementWg.Wait()` unblock
(`cmd/main.go:356-357`). 배치 도입으로 새로 생긴 경로는 "drain 도중 채널이
닫히는 경우" 하나뿐이며, 이는 Task 4에서 추가한
`TestCollectTradeBatchReportsClosedChannel`(`cmd/main_test.go:466`)이 정확히
그 케이스(닫히기 전 큐에 남아있던 이벤트까지 포함해 배치로 반환하는지)를
커버한다. 도미노는 Task 1-4 변경 후에도 그대로 성립한다.

**크래시 내구성**: 배치 트랜잭션이 커밋되기 전에 크래시가 나면, 그 배치에
속한 outbox 행은 전부 PENDING으로 남는다 — `SettleTradeBatch`의 실패/미완료
경로는 outbox 행을 트랜잭션 바깥에서 건드리지 않고, 유일한 마킹 지점인
`markSettledOutboxBatch`도 같은 트랜잭션 안에서만 실행되기 때문이다
(`internal/service/settlement_batch.go:323,333-350`). 다음 부팅의 리플레이
경로(단건 `SettleTrade` 경유, `OutboxReplayer`)가 PENDING 행을 재처리한다.
**A-3의 크래시 주입 테스트를 다시 돌릴 필요는 없다** — write-ahead 관문인
`OutboxWriter`(`internal/service/outbox_writer.go`)는 이 계획의 diff에서
전혀 손대지 않았고(브랜치 기준 `main` 대비 무변경 확인), 이번 변경은 이미
커밋된 outbox 이벤트를 "몇 건씩 묶어 소비하느냐"만 바꿨을 뿐 "언제·어떻게
커밋되느냐"는 바꾸지 않았다.

**성능 측정은 이 계획 범위 밖**: "나중에 일괄" 방침대로 GCP k6 측정은 다음
측정 사이클에서 B-1(B-1a/B-1b)과 묶어 hold 프로파일 same-session A/B로
수행한다. 그때 `settlement_batch_size` 히스토그램으로 실제 배치 크기 분포도
함께 확인한다(1에 몰려 있으면 그룹커밋이 발동하지 않는 부하라는 뜻). 이
문서는 어떤 처리량 수치도 주장하지 않는다 — 이번 계획에서는 측정하지 않았다.
