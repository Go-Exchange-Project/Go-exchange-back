# 14. 2차 리팩토링 ④ 입장 거절 품질 — 완료

- **완료일**: 2026-07-22
- **설계 문서**: [④ 입장 거절 품질 설계](../superpowers/specs/2026-07-21-admission-reject-quality-design.md)
- **구현 계획**: [④ 입장 거절 품질 구현 계획](../superpowers/plans/2026-07-21-admission-reject-quality.md)
- **근거**: ①②가 이미 "매달림 0, 빠른 503" 하드 보장을 delivered — ④는 그 거절의 **품질**만 더한다.

## 왜 필요한가 (그리고 왜 작은가)

과부하 시 매달림 없이 빠르게 거절한다는 하드 보장은 이미 ①(엔진 유입 게이트 +
바운디드 핸드오프)·②(코디네이터 입력 만석 시 논블로킹 503)가 제공한다. 두 경로
모두 `SERVICE_UNAVAILABLE`(503)을 반환하므로, 모든 과부하 경로가 유한 시간에
503을 낸다는 것은 이미 실증돼 있다. ④는 "매달림을 없앤다"가 아니라 **거절 품질
두 가지**만 더한다:

1. **Retry-After 부재**: 기존 503엔 백오프 힌트가 없어 클라이언트가 즉시
   재시도할 수 있다 → 과부하 증폭(retry storm).
2. **셰딩 관측 부재**: 과부하에서 얼마나·어디서 거절했는지 측정할 카운터가 없어
   ⑤(23번)가 "목표 초과 시 우아한 셰딩"을 실증할 수 없다.

**의도적으로 하지 않은 것**(설계 문서에 명시): ①의 엔진 게이트와 ②의 코디네이터
입력을 단일 admission controller로 통합하지 않는다 — 서로 다른 자원·용량을 보호하므로
단일 정책이 맞지 않고, 통합은 추상화만 늘린다. 히스테리시스도 붙이지 않는다 —
단순 워터마크로도 초과분만큼 자기 조절되며, ⑤에서 flapping이 실제 문제로
관측되면 그때 조건부로 붙인다.

## 어떻게 해결했나

**Task 1 (`67aec8f`+`6cf5cdc`)**: `internal/httpapi/response.go`의 `WriteError`/
`AbortWithError`에서 응답 상태가 503일 때만 `Retry-After: 1` 헤더를 중앙에서
세팅하는 `setRetryAfterForOverload` 헬퍼 추가. 헤더는 `c.JSON`/
`c.AbortWithStatusJSON`(바디 쓰기) **앞에** 세팅 — Go net/http는 바디 쓰기 후
헤더 설정을 무시하므로 순서가 정합성이다. 두 함수 모두에 직접 테스트를 붙여
(첫 리뷰에서 `AbortWithError` 쪽 커버리지 누락이 지적돼 후속 커밋으로 보완)
503만 헤더를 받고 다른 상태는 영향받지 않음을 확인했다. 중앙화이므로 현재의
두 503 경로(①게이트·②코디네이터)와 향후 모든 503이 자동으로 헤더를 받는다.

**Task 2 (`f4bce32`)**: `internal/metrics/metrics.go`에
`orders_admission_rejected_total{stage}`(`CounterVec`) 추가, stage 3종을
정확히 구분해 각 거절 지점이 자기 stage를 `Inc()`:
- `engine_gate` — `OrderService.CreateOrder`의 pre-DB 게이트(`IsIntakeAdmissible`
  false).
- `engine_handoff` — `CreateOrder`의 바운디드 핸드오프 타임아웃(`TrySubmitOrder`
  false → 보상).
- `coordinator` — `HoldCoordinator.Submit`의 입력 만석(논블로킹 `default`).

`order_service.go`에 `metrics` 임포트를 새로 추가했고(기존 미사용),
`hold_coordinator.go`는 이미 다른 메트릭에서 `metrics`를 쓰고 있어 `Inc()` 한
줄만 추가했다. **로직은 완전히 무변경** — 거절 조건·임계값·에러 메시지·보상
로직 어느 것도 건드리지 않았다.

## 결과

- **Retry-After**: 503 응답에 헤더가 실림(`TestWriteErrorSetsRetryAfterOn503`·
  `TestAbortWithErrorSetsRetryAfterOn503`), 비-503엔 실리지 않음
  (`TestWriteErrorNoRetryAfterOnNon503`)을 단위 테스트로 확인.
- **셰딩 메트릭**: 신규 카운터 단위 테스트(`TestOrdersAdmissionRejectedTotalIncrementsPerStage`)
  + 기존 세 거절 통합 테스트(`TestIntegrationCreateOrderFastRejectsWhenIntakeSaturated`·
  `TestIntegrationCreateOrderCompensatesWhenHandoffTimesOut`·
  `TestHoldCoordinatorSubmitReturnsUnavailableWhenInputFull`)에 before/after
  델타(+1) 단언을 추가해 각 stage가 정확히 자기 지점에서만 증분됨을 확인
  (카운터가 테스트 바이너리 전역이라 절대값이 아닌 델타로 검증).
- `go build`/`go vet`/전체 스위트(SKIP 0)/`-race`(service·cmd) 전부 그린.
  **기존 ①②③·정산·부트스트랩 통합 테스트 무수정 그린** — 거절 동작(503 반환·
  보상)이 완전히 불변이라는 증거.
- 리뷰: Task 1은 첫 리뷰에서 `AbortWithError`의 직접 테스트 누락(Important)이
  지적돼 대칭 테스트를 추가한 뒤 재검토 승인. Task 2는 sonnet 리뷰에서 세
  Inc() 위치·라벨 철자·델타 단언 패턴을 diff에서 직접 추적해 확인, Critical/
  Important 0건.

**이 조각은 셰딩률/처리량 수치를 주장하지 않는다** — 실제 과부하에서 재시도
폭풍이 완화되는지, stage별 셰딩 분포가 어떤지는 ⑤(23번 벤치마크)에서
`orders_admission_rejected_total{stage}` 분포로 측정한다. ④의 성공 기준은
"Retry-After 실림 + 메트릭 증분 + 회귀 그린"까지이며, 이 문서가 그것을 보인다.

## API 계약 영향 (프론트 — 최소)

`POST /orders`의 성공 응답은 불변. 신규는 **503 응답에 `Retry-After: 1` 헤더가
추가**된다는 점뿐 — 응답 바디 형식·에러 코드는 그대로다.

## 남은 것

없음 — ④는 Task 1-3 전부 완료. 이로써 **2차 리팩토링(가용성 100%)의 코드가
모두 완성**됐다 — ①(접수-매칭 분리)·②(자금 홀드 그룹커밋)·③(취소 우선
스케줄링)·④(입장 거절 품질). 남는 것은 ⑤(23번 GCP 실증)뿐: 스파이크에서 취소
실패율·주문 접수 p95·`orders_admission_rejected_total{stage}` 셰딩 분포를
①②③④ 종합으로 측정한다. 두 게이트 통합과 히스테리시스는 ⑤에서 실측 근거가
나오면 조건부로 재검토한다.
