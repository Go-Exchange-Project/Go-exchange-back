# 9번째 테스트 (2026-07-09): 매칭엔진 채널 길이로 병목 메커니즘 확인

## 커밋 해시

`a3d1226` (feat(monitoring): 매칭엔진 채널 길이 대시보드 패널 추가) — 관련 코드 커밋은 `3df33f2`(게이지 등록 함수), `92875da`(main.go 배선)

## 왜 이 테스트를 했는지

`internal/matching/engine.go`의 매칭엔진은 심볼 상관없이 **단일 `for-select` 루프** 하나로 모든 주문을 처리하고, `Match()` 안에서 체결이 나면 `me.ExecutionCh <- ...`(정산용)와 `me.SnapshotCh <- ...`(웹소켓 브로드캐스트용)를 이 루프 안에서 **동기적으로** 전송한다. 이 두 채널의 소비자(정산 워커, 브로드캐스트 고루틴)가 느려서 버퍼(`ExecutionCh`=1024, `SnapshotCh`=256)가 가득 차면, 이 전송 자체가 블로킹돼서 매칭 루프 전체가 멈추고 모든 심볼의 모든 주문이 줄줄이 밀린다는 가설을 세웠다. `08번 테스트`에서 확인한 매칭 지연 p95(14.2~27.5초)가 이 메커니즘 때문인지 실측으로 확인하려고, 채널 길이를 Prometheus 게이지로 노출했다.

## 왜 이 방식을 선택했는지

`docs/superpowers/specs/2026-07-09-matching-engine-channel-length-metrics-design.md`에서 결정한 대로, `promauto.NewGaugeFunc`로 5개 채널(`OrderCh`, `CancelCh`, `SnapshotReq`, `ExecutionCh`, `SnapshotCh`) 길이를 노출했다 — 별도 폴링 고루틴 없이 Prometheus가 스크레이프할 때마다 `len()`을 읽는 방식이라 코드가 단순하고, 기존 `MatchLatencyObserver` 콜백 패턴을 그대로 따라 `metrics` 패키지가 `matching` 패키지를 몰라도 되게 했다.

## 핵심 결과: 가설이 정확히 맞았다

### 채널 길이 시계열 (30초 간격, 램프업~정상상태 구간)

| 경과 시간 | `execution` 채널 | `order` 채널 | `snapshot` 채널 |
|---|---|---|---|
| 0~4분 | 0 | 0 | 0 |
| ~4.5분 (VU 87→100 부근) | 30 → 532 (차오르기 시작) | 0 (아직 정상) | 0 |
| ~5분 이후 (VU 100+) | **1024 (완전 포화)** | **1024 (완전 포화)** | 0 |
| 이후 부하 지속되는 내내 | **1024 유지** | **1024 유지** | 0 |
| 부하 종료 시점 | 0으로 즉시 복귀 | 0으로 즉시 복귀 | 0 (변화 없음) |

**`execution` 채널이 먼저 버퍼 상한(1024)까지 차오르고, 그 직후 `order` 채널도 뒤따라 상한까지 찬다. `snapshot` 채널은 테스트 내내 단 한 번도 0을 벗어나지 않는다.**

### 같은 시점의 매칭 지연 p95

VU~150~152 구간에서 `execution`/`order` 채널이 둘 다 1024로 포화된 상태일 때, `order_pipeline_match_latency_seconds` p95는 **14.75초**였다 — `08번 테스트`에서 관측한 값과 일치.

## 해석

1. **가설이 정확히 맞았다.** 매칭 지연의 진짜 메커니즘은 이거다: 정산(DB 쓰기) 워커가 체결 이벤트를 처리하는 속도가 주문 유입 속도를 못 따라가서 `ExecutionCh`(버퍼 1024)가 가득 찬다 → 매칭 루프 안의 `me.ExecutionCh <- ...` 전송이 블로킹된다 → 매칭 루프 자체가 멈춘다 → 매칭 루프가 못 비우니 `OrderCh`(버퍼 1024, 주문 입력 큐)도 가득 찬다 → 새 주문이 매칭엔진에 들어가지도 못하고 대기한다.
2. **`snapshot` 채널은 절대 범인이 아니다.** 웹소켓 브로드캐스트 고루틴은 부하가 아무리 몰려도 계속 잘 따라갔다(항상 0) — 4번 테스트에서 "매칭 로직 자체는 안 느리다"고 확인한 것과 마찬가지로, 이번엔 "브로드캐스트도 안 느리다"는 걸 확인했다. **병목은 오직 정산(DB 쓰기) 경로 하나다.**
3. **왜 정산이 느린가는 05번 테스트와 연결된다.** DB 커넥션 풀은 이미 튜닝했지만(05번), 정산 고루틴이 **단 하나**뿐이라는 점(`cmd/main.go`의 `go func() { for event := range me.ExecutionCh { ... } }()`)이 진짜 원인일 가능성이 크다 — 커넥션 풀이 25개까지 열려도 정산 자체가 순차 처리(단일 고루틴)라 병렬로 못 쓴다.
4. **다음 조치가 명확해졌다.** 막연한 "CPU 스케줄링 경쟁"이나 인프라 확장이 아니라, **정산 소비자를 여러 워커로 늘리는 것**(가장 간단한 코드 수정) 또는 **정산을 Redis Streams 같은 진짜 비동기 큐로 옮겨 여러 워커가 병렬 소비하게 하는 것**(로드맵의 Redis 도입 단계)이 직접적인 해결책이다.

## 이력서/포트폴리오용 요약 문장

- "매칭엔진의 내부 채널 길이를 Prometheus 게이지로 노출해, '매칭 지연 14~27초'의 진짜 메커니즘이 CPU 경쟁이 아니라 **단일 정산 워커가 체결 이벤트 처리 속도를 못 따라가 채널 버퍼가 포화되고, 그로 인해 매칭 루프 자체가 블로킹되는 백프레셔 문제**임을 정량적으로 확인했다."

## 다음 작업 제안

1. **가장 간단한 수정**: 정산 소비자 고루틴을 여러 개(워커 풀)로 늘려서 `ExecutionCh` 소비 속도를 높인다 — 코드 몇 줄로 이번 병목을 바로 해소할 수 있는지 먼저 확인.
2. **로드맵 2단계(Redis)로 자연스럽게 연결**: 워커 풀만으로 부족하면, 정산을 Redis Streams로 옮겨 재시작에도 안전하고 수평 확장 가능한 큐로 만든다.
3. 이번 수정 후 다시 04~09번과 같은 조건으로 재측정해서, 매칭 지연 p95가 실제로 얼마나 줄어드는지 확인한다.
