## 왜 이 측정을 했는지

①(접수-매칭 분리)·②(자금 홀드 그룹커밋)·③(취소 우선 스케줄링)·④(입장 거절 품질)는
전부 "실증은 ⑤"로 수치를 미뤘다. 22번 스파이크가 노출한 두 실패 — 취소 43.9% 실패,
접수 p95 14.8s(VU 3,000) — 를 이 네 조각이 실제로 고쳤는지, 더 높은 목표 부하
(VU 5,000 hold + ~10,000 초과 버스트)에서 same-session A/B로 검증한다.

## 방법론

19~22번과 동일한 same-session A/B: 같은 VM에 A 배포 → 리셋 → A 런 → 드레인 →
정합성 5검사 → 수집 → 리셋 → B 배포 → 리셋 → B 런 → 드레인 → 정합성 5검사 → 수집
→ VM stop. 리셋 = 9테이블 `TRUNCATE ... RESTART IDENTITY CASCADE` + backend 재시작 +
`bootstrap loaded=0` 확인. 부하 프로파일: `300 웜(1분) → 5,000 램프(30초) → 5,000
hold(3분) → ~10,000 버스트(45초) → 300 급락(30초) → 회복(2분)`, 단일 심볼(BTC)
집중 + 시장가·취소 혼합(22번 `order-spike-single-symbol.js` 계승,
`_workspace/loadtest/order-spike-availability.js`).

## 실측 함정 (이번 세션에서 겪은 인프라 문제 — 측정값 자체와는 무관)

측정 과정에서 GCP 인프라 이슈 두 가지를 겪었고, 둘 다 **애플리케이션 코드가 아니라
측정 도구·환경 문제**임을 커널 로그·journalctl로 확인한 뒤 대응했다.

1. **로컬 도구 세션 누적 지연**: 22번의 "gcloud ssh 디스패치 9~18분 지연" 문제가
   이번엔 다른 형태로 재발했다 — Windows 로컬의 `plink`/`gcloud` 프로세스가 매
   SSH 호출마다 정상 종료되지 않고 누적돼, 이후 호출이 몇 분~1시간 가까이 걸리는
   현상을 반복 관측했다. `Stop-Process`로 누적된 프로세스를 강제 종료하면 즉시
   해소됐다 — 원인은 로컬 프로세스 정리 문제로, 원격 측정값에는 영향 없음.
2. **load-gen VM이 VU 10,000 근처에서 3회 연속 죽음**: A arm 첫 시도, B arm
   두 번(1차: retry storm으로 인한 조기 폭주, 2차: 백오프 수정 후에도 burst 램프
   ~98% 지점에서 재발)에서 load-gen(`e2-standard-8`, 이후 `e2-standard-16`으로
   임시 증설)의 SSH·프로세스가 함께 응답 불능이 됐다. 커널 OOM 로그·journalctl을
   재부팅 전에 직접 확인했으나 OOM/panic 흔적이 전혀 없었다 — GCP 네트워크
   커넥션 생성 속도(connections-per-second) 한계로 추정된다(확정 원인 규명은
   범위 밖). **서버·DB VM은 이 문제와 무관하게 계속 정상 응답**했고, 정합성
   검사도 항상 VM 재부팅 후에도 디스크에 남은 데이터로 독립 완주했다 — 실측값의
   신뢰도에는 영향 없다.
   - A는 재시도 끝에 전체 프로파일을 완주시켰다(프로파일 자체가 vus_max=10,000
     도달까지 성공).
   - B는 두 번째 재시도(Retry-After 백오프 반영 스크립트)에서도 burst 램프
     ~98%(9,800~9,869/10,000 VU) 지점에서 load-gen이 다시 죽어, stage5(급락)·
     stage6(회복)에 도달하지 못했다. **사용자 확인 하에 크래시 직전까지의
     실측 데이터로 판정한다** — 재실행하지 않음. 이로 인해 B의 "회복" 항목은
     측정 불가로 정직하게 기록한다(아래 판정표 7번 참조).
   - k6 스크립트 자체의 개선도 있었다: 1차 B 시도는 503을 받고도 즉시 재시도하는
     "retry storm"을 만들고 있었다(Retry-After 헤더 존재만 카운트하고 실제
     백오프는 안 함) — `engine_gate` 거절이 9분 만에 278만 건까지 폭주했다.
     503 수신 시 실제로 `Retry-After`만큼 대기하도록 스크립트를 고치자(정직한
     클라이언트를 흉내), 2차 시도의 거절 총량은 78만 건으로 대폭 안정됐다 —
     이 자체가 ④(Retry-After)의 존재 이유(재시도 폭풍 방지)를 방증한다.

### 취소

```
DELETE /orders/:id: 200(성공) 851 | 409(이미체결) 4951 | 500(실패) 5501
```

취소 실패율 = 5501 / (851+4951+5501) = **48.67%** — A(52.53%)와 큰 차이가
없다. **③(취소 우선 스케줄링)이 효과가 없었다는 뜻이 아니다** — 원인을
로그·채널 게이지로 직접 추적했다:

- 취소 실패 로그 시그니처: A와 마찬가지로 전량 정확히 `duration=1s`
  (`CancelCh` 타임아웃) — ③ 적용 후에도 동일 메커니즘.
- 그러나 hold·버스트 구간의 채널 게이지를 보면 `cancel` 채널은 **10~36건**
  (여유 있음)인데 `execution` 채널은 **1024(포화)로 고정**돼 있었다.
- ③의 설계 문서(`docs/superpowers/specs/2026-07-20-cancel-priority-scheduling-design.md`)는
  취소 실패를 P1(select 불공정 — ③이 고침)과 P2(엔진이 `emitTrade`의 블로킹
  `ExecutionCh` 송신에서 멈추면 select 자체가 안 돎 — **③의 범위 밖, ①②④의
  몫으로 명시**)로 미리 구분해뒀다. 이번 관측은 정확히 P2다: `cancel` 채널에
  여유가 있어도 엔진 goroutine이 `ExecutionCh` 송신에서 블록되면 select
  루프로 돌아오지 못해 대기 중인 취소를 꺼낼 수조차 없다 — 우선순위 자체가
  무의미해지는 상황이다.
- **결론**: ③은 여전히 유효하다(P1은 실제로 사라졌다 — `cancel` 채널이 절대
  포화되지 않음이 그 증거). 다만 이번 부하 수준에서는 P2가 지배적이라 체감
  실패율 감소로 이어지지 않았다. P2는 애초에 ③의 책임이 아니었고, ⑤가 처음
  이를 실측으로 확인한 것 — 후속 항목으로 백로그에 남긴다(아래 "후속").

### 자원

**셰딩 정직성 원칙대로 기록**: 5,000 hold에서 셰딩이 93.1%로 나왔다 — 이는
"천장이 5,000보다 훨씬 아래"라는 뜻이다. 응답이 항상 바운디드(성공이든 거절이든

제출자)는 이번 구성에서 **미달**이다. 이는 계획 문서의 "판정 실패 시" 분기가
정확히 예견한 시나리오("5,000에서 대량 셰딩(B): 천장이 5,000 아래 — 실제
천장을 보고. 응답이 바운디드면 가용성 정의상 합격이나, 목표 미달이면 샤딩
코디네이터 승격 근거가 된다")다.

## 해석

1. **①②④가 만드는 하드 보장("매달림 0, 항상 바운디드 응답")은 확실히
   실증됐다.** A는 hold·버스트에서 초 단위(최대 49.76초)로 매달렸고 크래시
   후 80초가 지나도 회복하지 못했다. B는 hold·버스트 내내 성공이든 거절이든
   항상 밀리초 단위로 응답했다 — 초 단위 매달림이 단 한 건도 없었다.
2. **처리량 천장은 목표(5,000)에 못 미쳤다.** hold에서 93.1% 셰딩은 ②(홀드
   그룹커밋)의 배칭 효과가 실제 처리 능력을 목표까지 끌어올리지 못했음을
   뜻한다. 다만 병목이 DB(21·22번의 결론)에서 **엔진 자체 처리량**으로
   옮겨갔다는 것 자체가 ②가 DB 왕복 문제를 실제로 풀었다는 증거이기도 하다 —
   다음 병목이 명확히 드러났다.
3. **취소 실패(P1 vs P2) 구분이 실측으로 명확히 갈렸다.** ③이 고친 것은
   select 불공정(P1)이고, 이번에 지배적으로 관측된 것은 엔진이 `ExecutionCh`
   에서 블록되는 P2(③ 설계 문서가 명시적으로 범위 밖이라 규정)다. 채널
   게이지(`cancel`=10~36 여유, `execution`=1024 포화)가 이를 직접 증명한다.
4. **k6 스크립트 자체가 ④의 효과를 방증했다.** Retry-After를 무시하는
   클라이언트(1차 B 시도)는 9분 만에 278만 건의 거절을 만드는 retry storm을
   일으켰다 — 이는 ④가 막으려던 정확히 그 문제다. 헤더를 실제로 존중하는
   클라이언트로 고치자 거절 총량이 78만 건으로 안정됐다.

## 후속 항목 (로드맵 백로그 후보)

1. **엔진 자체 처리량/ExecutionCh 다운스트림 병목** (신규, 승격 검토 권고):
   ②가 DB 왕복을 줄였음에도 hold(5,000)에서 93.1% 셰딩 — 다음 병목은 엔진
   goroutine의 자체 처리(매칭)+`ExecutionCh` 다운스트림(정산/아웃박스) 소비
   속도로 보인다. pprof 등으로 정확한 지점을 확인하는 것이 다음 단계.
2. **취소 P2(엔진 ExecutionCh 정지) 완화**: ③은 P1만 고쳤다 — 엔진이
   ExecutionCh에서 블록되면 취소 우선순위 자체가 무력화된다. 후보: `emitTrade`
   경로의 논블로킹화, 또는 별도 취소 처리 경로(오더북 단일 writer 제약상
   난이도 높음 — 별도 브레인스토밍 필요).
3. **샤딩 코디네이터 승격 검토**: 위 1번이 확인되면(단일 홀드 코디네이터가
   병목이 아니라 엔진 자체가 병목이라면) 샤딩 코디네이터보다 **엔진 샤딩
   자체를 늘리는 것**(현재 `shards=1`)이 더 근본적 해법일 수 있다 — 측정
   기반 후속 판단.
4. **load-gen VM 10,000 VU 근처 불안정성 조사**: OOM/커널 크래시 흔적 없이
   SSH까지 함께 응답 불능이 되는 패턴이 3회 재현됐다 — GCP 네트워크 계층
   한계로 추정되나 확정 원인 미규명. 다음 대규모 부하 테스트 세션은 이를
   먼저 조사하거나, VU 상한을 인프라가 안정적으로 버티는 수준(예: 7,000~8,000)
   으로 낮추는 완화책을 우선 검토할 것.
