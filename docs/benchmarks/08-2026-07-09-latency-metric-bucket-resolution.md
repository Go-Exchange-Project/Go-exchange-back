# 8번째 테스트 (2026-07-09): 매칭/정산 지연 지표 버킷 해상도 수정 결과

## 커밋 해시

`6dfced6` (fix(metrics): 매칭/정산 지연 히스토그램 버킷을 60초까지 확장)

## 왜 이 테스트를 했는지

`04~07`번 테스트에서 `order_pipeline_match_latency_seconds`의 p95가 매번 정확히 10초로 고정된 게 이상했다. `docs/superpowers/specs/2026-07-09-latency-metric-bucket-resolution-design.md`에서 원인을 확인했다 — 이 히스토그램이 `prometheus.DefBuckets`(최대 10초)를 써서, 실제 지연이 10초를 넘으면 전부 `+Inf` 버킷에 뭉뚱그려지고 `histogram_quantile`이 10초로 잘라서 보여준 것이었다. 버킷을 60초까지 확장해서 **진짜 값**을 확인했다.

## 왜 이 방식을 선택했는지

`internal/metrics/metrics.go`의 `Buckets` 설정만 바꾸면 되는 최소 변경이라, 코드 수정 후 바로 같은 조건으로 재측정해서 진짜 수치를 확인하는 게 가장 빠르고 확실한 검증 방법이었다.

## 환경 (04~07번 테스트와 동일 조건)

- 서버 인스턴스: GCP `e2-medium`(2 vCPU, 4GB), 서울 리전

## 해석

1. **가설이 맞았다.** `order_pipeline_match_latency_seconds`는 진짜로 히스토그램 버킷 상한(10초)에 눌려서 축소 보고되고 있었다. 실제 매칭 큐잉+처리 지연은 VU가 높아질수록 **14초에서 최대 27.5초**까지 벌어졌다 — 지금까지 문서화했던 "10초"보다 최대 2.75배 심각한 상황이었다.
2. **`order_settlement_duration_seconds`는 버킷 문제가 없었다(0.17초).** 즉 지연의 원인은 정산(DB 쓰기) 단계가 아니라, 매칭 자체(또는 매칭 큐 대기)에 있다 — `06`번 순수 벤치마크에서 확인했듯 매칭 로직 자체는 마이크로초 단위로 빠르므로, 이 지연은 여전히 "매칭 고루틴이 실행 순서를 못 받아 큐에서 대기하는 시간"으로 해석하는 게 맞다.
3. **7번(CPU 핀닝) 문서를 다시 봐야 한다.** 그 실험에서 "매칭 지연 10초, 변화 없음"이라고 기록했는데, 이제 보니 그 "변화 없음"도 버킷 상한 때문에 실제 변화를 못 본 것일 수 있다 — CPU 핀닝이 매칭 지연에 진짜로 영향을 줬는지 안 줬는지는 이 지표가 고쳐지기 전이라 정확히 판단할 수 없었던 것으로 재해석해야 한다.
4. **다음 조사 방향이 명확해졌다.** 진짜 병목은 "매칭 고루틴이 큐에서 최대 27초씩 대기하는 것"이다. 이건 여전히 CPU 스케줄링 경쟁 가설과 일치하지만, 이번엔 정확한 수치를 가지고 재검증할 수 있다.
