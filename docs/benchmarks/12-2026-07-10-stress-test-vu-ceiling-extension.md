# 12번째 테스트 (2026-07-10): VU 상한 확장(800→3,000)으로 진짜 벽 재확인

## 커밋 해시

`0c1df36` (docs: 스트레스 테스트 주석의 VU 상한 수치를 800에서 3000으로 수정) — 애플리케이션 코드는 `d50e383`(정산 워커 풀) 이후 변경 없음. 서버는 `e2-highcpu-4`(11번 테스트에서 확정) 그대로 사용.

## 왜 이 테스트를 했는지

`11-2026-07-09-vm-cpu-scaling-control.md`에서 VU 700+ 구간부터 CPU가 다시 오르는 걸 봤는데, 당시 k6 스크립트가 VU 800에서 램프업을 끝내도록 짜여 있어서 이게 진짜 시스템 한계인지 스크립트 설정 때문인지 구분이 안 됐다. 실제 거래소는 급등/급락 시 매우 짧은 시간에 훨씬 많은 동시 접속이 몰릴 수 있다는 지적에 따라, VU 상한을 3,000까지 올려서 진짜 벽이 어디인지 재확인했다.

## 왜 이 방식을 선택했는지

`docs/superpowers/specs/2026-07-09-stress-test-vu-ceiling-extension-design.md`에서 결정한 대로, 기존 램프업 패턴(대략 2배씩 증가)을 이어서 1,600/3,000 단계를 추가했다. 이 과정에서 부하생성기(`e2-small`, RAM 2GB)가 VU~584에서 OOM으로 먼저 죽는 걸 발견해서, 부하생성기도 `e2-standard-2`(RAM 8GB)로 함께 리사이즈한 뒤 재실행했다 — 서버가 아니라 부하생성기 자체가 병목이 되면 측정이 무의미해지기 때문이다.

## 환경

- 서버: `e2-highcpu-4`(4 vCPU, 4GB) — 11번 테스트에서 확정된 사양 그대로
- 부하생성기: `e2-small`(2 vCPU, 2GB) → **`e2-standard-2`(2 vCPU, 8GB)로 리사이즈** (1차 시도가 OOM으로 실패해서)
- k6 `STRESS_STAGES`: 50→100→200→400→800→**1,600→3,000**(각 2분), `TOTAL_USERS=3000`

## 실행한 정확한 커맨드

```bash
k6 run -e BASE_URL=http://10.10.0.3:8080 -e DEV_TOOLS_TOKEN=<토큰> loadtest/order-submission-stress.js
```

## 1차 시도: 부하생성기 OOM (측정 실패, 그 자체로 유의미한 발견)

`e2-small`(RAM 2GB) 그대로 실행했을 때, VU~584 지점에서 k6 프로세스가 리눅스 OOM killer에 의해 강제 종료됐다:

```
Out of memory: Killed process 11562 (k6) total-vm:3062056kB, anon-rss:1732900kB, ...
```

**서버가 아니라 부하생성기 쪽이 먼저 무너진 것** — 이 자체가 "부하생성기 사양도 실측 대상"이라는 걸 보여주는 발견이라 기록해둔다.

## 2차 시도(부하생성기 리사이즈 후): 완주, 진짜 벽 확인됨

**시스템은 VU 3,000까지 죽지 않고 완주했다** — 다만 성능은 눈에 띄게 나빠졌다(p95 7.05초, 최댓값 32.54초 — 11번 테스트의 p95 1.32초보다 훨씬 나쁨).

### 시계열로 본 진짜 벽의 위치

| 구간 | CPU | `order` 채널 | `execution` 채널 | 매칭 지연 p95 |
|---|---|---|---|---|
| VU 낮음(초반) | 13~27% | 0 | 0 | ~0.005초 |
| **VU ~700~800 도달** | **90% 근처로 평탄화** | **1024로 포화 시작** | 0(끝까지) | 급상승 시작 |
| VU 800~1600 구간 | 90% 유지 | 1024 유지 | 0 | 4.9~9.75초 |
| VU 1600~3000 구간 | 90% 유지(더 안 오름) | 1024 유지 | 0 | **14.7초까지 계속 상승** |
| 부하 종료 후 | 1~2%로 즉시 복귀 | 0 | 0 | (데이터 없음, 트래픽 없음) |

## 해석

1. **진짜 벽은 VU 700~800 부근이 맞았다.** 11번 테스트에서 관찰한 "VU700+부터 CPU가 다시 오른다"는 게 스크립트 설정 때문이 아니라 진짜 시스템 한계였음을 확인했다.
2. **하지만 "벽"의 의미가 예상과 달랐다 — 시스템이 죽는 게 아니라 큐잉 지연이 계속 쌓이는 것이다.** CPU가 90%에서 평탄화된 뒤로는 VU를 3,000까지 올려도 CPU는 더 안 오르는데, 지연은 4.9초→9.75초→14.7초로 계속 늘어난다. 이건 교과서적인 큐잉 이론 그대로다 — 처리 용량(CPU)이 한계에 도달하면, 유입량이 늘어날수록 대기 줄이 계속 길어질 뿐 시스템이 크래시하지는 않는다.
3. **정산(execution) 쪽은 VU 3,000까지도 전혀 문제없었다.** `execution` 채널이 끝까지 0이었다는 건, 10번 테스트에서 도입한 정산 워커 풀이 이 정도 규모에서도 확실히 버틴다는 뜻이다.
4. **부하생성기 자체도 실측 대상이었다.** `e2-small`(RAM 2GB)로는 VU 600 근처도 제대로 측정 못 하고 죽었다 — 앞으로 더 큰 규모의 테스트를 하려면 부하생성기 사양도 같이 고려해야 한다는 교훈을 얻었다.

## 다음 작업 제안

1. VU 700~800이라는 진짜 처리 용량 한계를 늘리려면, 지난번에 논의한 대로 Postgres를 별도 인스턴스로 분리하거나 매칭엔진을 심볼별로 병렬화(샤딩)하는 게 다음 후보다.
2. `http_req_failed`가 이번에도 1.16%(3000/257032)로 나왔다 — 계속 관측되는 소량의 네트워크 레벨 실패 원인을 아직 조사하지 않았다.

## 범위 밖 (Out of Scope)

- Postgres 분리, 매칭엔진 샤딩의 실제 구현 — 별도 브레인스토밍.
- `http_req_failed` 원인 조사 — 별도 작업.
