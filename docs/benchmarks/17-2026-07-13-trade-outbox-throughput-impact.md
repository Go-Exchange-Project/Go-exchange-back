# 17번째 테스트 (2026-07-13): trade outbox(A-3) 도입 전/후 처리량 비교

## 커밋 해시

- **before**: `f2ec22a` (A-3 직전 — A-1/A-2 정산 파티셔닝, A-6 리컨실리에이션, C-2 백업까지 포함)
- **after**: `37a3e49` (feat(settlement): trade outbox 기반 write-ahead 정산으로 체결 유실 창 제거)

## 왜 이 테스트를 했는지

A-3에서 정산 경로에 write-ahead outbox를 넣어 크래시/배포 시 체결 유실 창을 없앴다.
그 대가로 정산 경로에 DB 쓰기가 추가됐으므로(outbox INSERT + 처리 후 MarkProcessed),
처리량/레이턴시에 얼마나 영향을 주는지 실측이 필요했다. A-3 완료 문서의 유일한
잔여 검증 항목이었다.

## 왜 이 방식을 선택했는지 (방법론)

- **같은 VM·같은 세션 연속 A/B**: 배포된 코드가 16번 당시(≈`62a6d21`, A-1 이전)
  구버전이라, 그걸 before로 쓰면 A/B 차이가 "A-3만"이 아니라 A-1/A-2/A-6/C-2까지
  누적된다. outbox의 순수 영향만 격리하려고 before를 **A-3 직전 커밋 `f2ec22a`로
  재배포**했다. 16번 문서 수치와의 직접 비교는 환경/세션이 달라 하지 않는다.
- **매 측정 전 빈 DB + backend 재시작**: 두 측정을 동일한 초기 조건(빈 오더북,
  cold 커넥션 풀)에서 시작. before/after 각각 TRUNCATE 후 재시작(bootstrap
  loaded=0 확인).
- **두 프로파일로 교차 검증**: k6 run-to-run 변동(16번에서 7.9% 차이를 노이즈일
  수 있다고 봤던 문제)을 배제하려고 서로 다른 부하 형태 2개로 각각 A/B:
  - **램프업**: VU 50→3000 (7단계 ×2분 = 14분). 포화점까지의 곡선.
  - **hold**: VU 1.5분 워밍업 → 3000에서 8분 유지. steady-state 포화 처리량.
    "포화점을 이미 아는데 매번 처음부터 올릴 필요가 있나"는 판단에 따라, 관심
    구간(3000 VU)에서 오래 관측하는 프로파일을 추가했다.
- 축약: TOTAL_USERS=3000, VU 상한 3000 (이전 테스트들에서 이미 이 구간이 포화).

## 실행한 정확한 커맨드

부하생성 인스턴스(`goexchange-stress-load-gen`)에서 서버 내부 IP 대상:

```bash
# 램프업
k6 run -e BASE_URL=http://10.10.0.3:8080 -e DEV_TOOLS_TOKEN=<토큰> ~/stress-vu3000.js
# hold
k6 run -e BASE_URL=http://10.10.0.3:8080 -e DEV_TOOLS_TOKEN=<토큰> ~/stress-hold3000.js
```

환경: 서버 `e2-medium`(2 vCPU), DB 별도 인스턴스(13번 이후 분리), load-gen
`e2-small`, k6 v2.0.0. 모두 서울 리전.

## 원인 규명: outbox INSERT가 아니라 MarkProcessed UPDATE가 주범

after hold 실행 직후 서버 메트릭:

```
trade_outbox_flush_batch_size_sum   44300   / _count 991   -> 평균 44.7건/배치
trade_outbox_flush_seconds_sum      6.21s   / _count 991   -> 평균 6.3ms/배치
trade_outbox_write_errors_total     0
order_settlement_duration_seconds_sum 504.5 / _count 44300 -> 평균 11.4ms/정산
```

- **outbox INSERT는 저렴하다**: 44,300건이 991개 배치로 뭉쳐(평균 44.7건/배치,
  64 상한에 근접) INSERT 왕복은 991회뿐. group commit(64건/5ms)이 설계대로 작동.
- **진짜 비용은 `MarkProcessed`다**: 정산이 끝난 뒤 각 이벤트를 PROCESSED로
  마킹하는 UPDATE가 **trade당 1회씩 44,300회 개별로** 나간다. 배치화돼 있지 않다.
  이 환경은 이전 테스트들(14~16번)에서 이미 DB/CPU가 병목이었으므로, 정산 트랜잭션
  바깥의 이 추가 왕복 44,300회가 곧바로 처리량 17% 저하로 반영됐다.

즉 "정합성(유실 0)을 산 대가로 처리량을 지불했다"는 결론은 맞지만, 그 대가의
대부분은 **최적화 여지가 있는** MarkProcessed 왕복이다.

## 최적화 제안 (다음 작업 후보)

1. **MarkProcessed를 정산 트랜잭션에 흡수** (가장 유력): 지금은 정산 트랜잭션
   커밋 → 별도 UPDATE로 outbox 마킹. 대신 정산 트랜잭션 안에서 해당 outbox 행의
   status를 함께 UPDATE하면 왕복 1회가 통째로 사라진다. 멱등성은 그대로(정산과
   마킹이 원자적이 되어 오히려 견고). trade당 왕복 2회 추가 → 1회로.
2. **MarkProcessed 배치화**: 워커가 처리한 outbox ID를 모아 `UPDATE ... WHERE id
   IN (...)`로 한 번에. 1번이 어려운 경로(예: 실패의 내구 기록만 남은 경우)에 보완.
3. **B-4 정산 그룹커밋과 함께**: 여러 체결의 정산을 한 트랜잭션으로 묶으면
   outbox 마킹도 자연히 그 트랜잭션에 합류. A-3의 이 비용은 B-4의 동기를 강화한다.

## 해석

1. **A-3의 처리량 대가는 실재하고(~17%), 재현 가능하다.** 정합성 목표상 받아들일
   수 있는 트레이드오프지만, 공짜가 아님을 실측으로 확인했다.
2. **그 대가는 대부분 제거 가능한 종류다.** outbox INSERT(배치화됨)가 아니라
   MarkProcessed 개별 UPDATE가 원인이므로, 위 1번(정산 트랜잭션 흡수)으로 상당 부분
   회수할 수 있을 것으로 보인다 — 별도 측정으로 확인할 가치가 있다.
3. **에러율은 두 빌드 모두 0%**였다. A-3가 정합성을 위해 추가한 경로가 정상 부하에서
   실패를 유발하지 않는다(graceful degradation은 레이턴시로만 나타남).
4. **레이턴시 절대값(p95 15~21s)은 VU 3000이 이 하드웨어에 과포화라 큰 것**이고,
   A/B 비교의 핵심은 절대 레이턴시가 아니라 상대 처리량이다.

## 범위 밖 (Out of Scope)

- 위 최적화(MarkProcessed 흡수/배치화)의 구현과 재측정 — 별도 작업. 로드맵상
  B-4(정산 그룹커밋)와 함께 다루는 것이 자연스럽다.
- 포화점 자체를 높이는 작업(모니터링 스택 분리, 인스턴스 확장 등)은 이번 스코프가
  아니다 — 고정 하드웨어에서 A-3의 상대 비용을 재는 것이 목적이었다.
