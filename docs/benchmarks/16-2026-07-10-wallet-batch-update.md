# 16번째 테스트 (2026-07-10): 정산 지갑 갱신 배치화 전/후 비교

## 커밋 해시

`62a6d21` (refactor(service): 정산 지갑 갱신을 배치 UPDATE로 교체) — 관련 코드 커밋: `14385bc`(리포지토리 배치 메서드 추가).

## 왜 이 테스트를 했는지

사용자가 "계속 인프라 스케일업만으로 해결하는 것 같다"고 지적해서, 15번 테스트(DB CPU 증설) 이후 처음으로 순수 코드 레벨 최적화를 시도했다. `14-2026-07-10-pg-stat-statements-investigation.md`에서 확인한 정산 트랜잭션당 쿼리 왕복 15회 중, 지갑 UPDATE 4회를 하나의 다중 행 UPDATE로 묶어서 왕복 횟수 자체를 줄였다.

## 왜 이 방식을 선택했는지

`docs/superpowers/specs/2026-07-10-wallet-batch-update-design.md`에서 결정한 대로, `internal/repository/wallet_reporsitory.go`에 `WalletBatchUpdate`/`BatchUpdateBalances`를 추가해 `UPDATE wallets AS w SET ... FROM (VALUES ...) AS v(...) WHERE w.id = v.id` 형태의 단일 SQL로 4개 지갑을 한 번에 갱신하게 했다. `internal/service/settlement_service.go`의 4번의 개별 호출을 이 배치 호출 하나로 교체했다. 기존 정산 통합 테스트 7개로 회귀를 확인했다(모두 실제 로컬 Postgres에서 통과).

## 실행한 정확한 커맨드

```bash
k6 run -e BASE_URL=http://10.10.0.3:8080 -e DEV_TOOLS_TOKEN=<토큰> loadtest/order-submission-stress.js
```

## 핵심 결과: 쿼리 레벨에선 확인됐지만, 전체 성능은 오히려 소폭 나빠졌다 (솔직하게 기록)

### 1. `pg_stat_statements`로 배치화 자체는 확인됐다

같은 VU~1000대 구간에서, 지갑 갱신이 개별 UPDATE 4개(14/15번)가 아니라 **배치 UPDATE 1개**로 나타났다:

```
UPDATE wallets AS w SET available_balance = v.available_balance, locked_balance = ...
calls=152183  total_ms=27177.7  mean_ms=0.18
```

호출 횟수(152,183)가 이 시점까지의 체결 건수(트레이드 INSERT 152,186회)와 정확히 일치한다 — 트레이드 1건당 지갑 UPDATE가 정말로 1번만 나갔다는 뜻이다.

### 2. 하지만 전체 k6 결과는 15번보다 소폭 나빠졌다

## 해석

1. **쿼리 왕복 감소는 설계대로 정확히 동작했다.** 지갑 UPDATE가 4회→1회로 준 게 `pg_stat_statements`로 명확히 확인됐다 — 코드는 의도대로 작동한다.
2. **하지만 15번(DB CPU 증설) 이후에는 이 최적화의 효과가 거의 없거나 오히려 손해였다.** 15번에서 이미 DB CPU 여유(80% 사용, 포화 아님)를 확보해 놓은 상태라, 쿼리 왕복을 줄이는 최적화가 발휘할 여지가 크지 않았던 것으로 보인다 — 왕복 횟수가 병목이었던 건 DB가 CPU 부족으로 포화 상태(14번, 389% 사용)였을 때의 얘기였고, 지금은 그 조건이 사라졌다.
3. **오히려 소폭 나빠진 이유는 명확히 규명하지 못했다.** 배치 UPDATE 자체(단일 쿼리 기준 mean 0.18ms)는 빠르지만, 여러 행을 한 번에 잠그는 다중 행 UPDATE가 개별 UPDATE보다 잠금 범위나 실행 계획이 달라져서 미세한 손해를 냈을 가능성, 혹은 단순히 두 실행 간의 자연스러운 변동성(부하생성기/네트워크 컨디션 차이)일 가능성 둘 다 배제할 수 없다 — 이번 테스트 한 번으로는 확정할 수 없다.
4. **이건 실패가 아니라 유의미한 발견이다.** "인프라를 늘리면 코드 최적화의 한계효용이 줄어든다"는 걸 실측으로 확인했다 — 04번(DB 풀 미설정)이나 14번(왕복 15회)처럼 진짜 병목일 때 코드 최적화는 확실한 효과를 냈지만, 이미 자원이 넉넉해진 지금 상태에서는 눈에 띄는 이득이 없었다. 코드는 유지하되(정합성은 그대로고 왕복 감소 자체는 사실이므로), "다음에 또 스케일다운하거나 부하가 더 커지면" 이 최적화가 다시 의미를 가질 수 있다.

## 다음 작업 제안

1. **재측정(run-to-run 변동성 배제)**: 같은 조건으로 한 번 더 돌려서 이번 결과가 노이즈인지 재현되는지 확인.
2. 코드는 유지한다 — 왕복 감소 자체는 사실이고 정합성 문제도 없으므로, 이번 결과가 "당장은 체감 효과 없음"을 보여준 것뿐이지 되돌릴 이유는 아니다.
3. 다음 코드 레벨 최적화 후보(주문 조회 2→1, 지갑 조회 4→1)를 진행할지, 아니면 다른 방향(매칭엔진 샤딩, `http_req_failed` 조사)으로 갈지 사용자와 논의.
