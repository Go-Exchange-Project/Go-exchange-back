# B-1c: 체결 브로드캐스트 배치화 설계

- **날짜**: 2026-07-16
- **상태**: 설계 검토 중
- **출처**: [19번 벤치마크](../../benchmarks/19-2026-07-16-b1-b4-batch-remeasurement.md) 해석 2 — B-1a가 스냅샷은 코얼레싱했지만 체결 브로드캐스트는 대상이 아니어서, 처리량이 5배 오르자 체결 fanout이 새로운 상대적 병목으로 부상(WS 300클라이언트에서 A −23.9% vs B −33.5%).

## 왜 필요한가 (측정 근거)

측정 4에서 서버는 초당 약 8.2만 건의 WS 메시지를 전송했다(300 클라이언트 × ~275 msg/s). 이 중 스냅샷은 심볼당 10/s로 캡핑돼 있으므로(B-1a), 대부분이 **체결마다 1건씩 나가는 trade 메시지**다. 메시지 1건 = hub 채널 송신 1회 + 클라이언트 300개 루프 + 클라이언트별 채널 송신·WS 프레임 write(syscall) 각 1회. 체결이 초당 수백 건이면 이 고정비용이 클라이언트 수 × 체결 수로 자란다.

한편 B-4 그룹커밋은 이 부하에서 **배치의 97%가 상한 32건**으로 커밋된다 — 즉 정산 워커의 손에는 이미 "방금 함께 커밋된 체결 32건"이 들려 있는데, 이걸 32번에 나눠 브로드캐스트하고 있다.

## 설계: 정산 배치 단위로 묶어 보낸다

**B-4 배치 = 브로드캐스트 배치.** 새 타이머·고루틴·버퍼 없음:

- `settleTradeBatchWithFallback`이 Applied trade들을 **심볼별로 그룹핑**해(워커 큐는 심볼 해시 파티션이라 한 배치에 여러 심볼이 섞일 수 있음) 심볼당 1건의 WS 메시지로 발행한다.
- 와이어 형식: `{"type":"trades","data":[<기존 trade 객체>, ...]}` — 배열 원소는 기존 `"type":"trade"`의 `data` 객체와 동일 필드. 단건 경로(폴백·리플레이·재시도)도 원소 1개짜리 같은 형식을 쓴다 — **`"type":"trade"`는 완전히 은퇴**시켜 형식을 하나로 유지한다.
- hub/ws 패키지는 **무변경**: 심볼 라우팅(B-1b)·클라이언트 관리 그대로. 변경은 `cmd/main.go`의 마샬 지점과 프론트엔드 핸들러뿐이다.

**효과 (측정 4 조건 추정)**: 고부하에서 trade 메시지 수가 ~32:1로 줄어 클라이언트별 채널 송신·프레임 write가 초당 ~8.2만 → 수천 건대로 떨어진다. 저부하에서는 배치=1이라 지금과 동일(추가 지연 0) — B-4 그룹커밋과 같은 적응 특성을 공짜로 물려받는다. 페이로드 바이트 총량은 거의 그대로다(데이터가 주는 게 아니라 메시지 경계가 줄어드는 것) — 목표는 대역폭이 아니라 fanout 고정비용(syscall·채널·루프)이다.

## 검토한 대안

- **시간 기반 코얼레싱(스냅샷처럼 100ms 버퍼)**: 새 버퍼·타이머·고루틴이 필요하고 지연이 추가된다. B-4 배치가 이미 같은 묶음을 공짜로 제공하므로 기각.
- **이중 형식(legacy는 per-trade 유지, 구독자만 배치)**: 호환 100%지만 hub 핫패스에 형식 분기가 생기고, 측정된 병목이 바로 legacy full-feed 시나리오였으므로 정작 병목이 안 풀린다. 기각.
- **hub 내부만 배치(와이어는 per-trade 유지)**: 채널 비용만 줄고 지배 비용(프레임 write syscall)이 그대로 남는다. 기각.
- **체결 집계(aggTrade식 동일가격 병합)**: 데이터를 요약(유실)하는 다른 상품이다. 배치는 원본 이벤트를 전부 보존하므로 우선하고, 집계 피드는 필요해지면 별도 항목.

## 호환성: 프론트 선배포로 끊김 제거

프론트(`Go-exchange-front/src/pages/Index.tsx`)는 `"type":"trade"`를 체결 내역에 소비하고, unknown type은 조용히 무시한다(크래시 없음). 배포 순서로 호환 문제를 없앤다:

1. **프론트 먼저**: `"trades"` 배열 핸들러 추가(기존 `"trade"` 핸들러는 유지 — 양쪽 형식 수용).
2. **백엔드 나중**: `"trades"`로 전환.

이 순서면 어느 시점에도 체결 피드가 끊기지 않는다. 역순이면 프론트 업데이트 전까지 체결 내역만 멈춘다(무시되므로 크래시는 아님).

## 검증 계획

1. 단위: 심볼 그룹핑(혼합 심볼 배치 → 심볼당 1메시지), 형식(`trades` + 배열), Applied 0건이면 발행 없음, 단건 경로도 같은 형식.
2. 기존 ws 패키지 테스트 전부 그대로 통과(hub 무변경의 증거).
3. 프론트: `trades` 배열이 체결 내역에 순서대로 반영되는지 수동 확인(로컬 백엔드 연결).
4. **GCP A/B는 다음 측정 사이클**: 다중 심볼 프로파일 작업(숙제 ②)과 묶어, 측정 4 조건(legacy×300) 재현으로 WS 부하 시 하락폭(−33.5%)이 줄어드는지 확인.
