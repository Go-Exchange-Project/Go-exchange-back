# ⑤ 가용성 스파이크 벤치마크 설계 (2차 리팩토링 · 23번 실증)

- **날짜**: 2026-07-22
- **상태**: 설계 검토 중
- **로드맵**: [2차 리팩토링 ⑤](../../refactor/README.md) — 가용성 100%의 마지막 조각(실증)
- **근거**: ①②③④는 코드로 완성됐고 전부 "실증은 ⑤"로 수치를 미뤘다. ⑤는
  22번 스파이크가 노출한 두 실패(취소 43.9%, 접수 p95 14.8s)를 ①②③④가 고쳤음을
  GCP에서 same-session A/B로 증명하는 **측정**이다. 코드 변경 없음.

## 왜 이 측정이 필요한가

[22번 벤치마크](../../benchmarks/22-2026-07-18-outbox-ab-and-spike.md)의 스파이크
시나리오(측정 4)가 두 가용성 실패를 노출했다:
- **취소 API 43.9% 실패**: VU 300→3000 급등 구간에서 `CancelCh`(버퍼 1024)가
  주문 커맨드로 포화 → 새 취소가 1초 타임아웃(`ErrCancelOrderTimedOut`, 500).
  정합성은 무해(409/500 분류 정확)했으나 가용성 실패.
- **접수 p95 14.8s**: hold(VU 3000) 구간 `POST /orders`가 엔진 소비/DB CPU에
  매달림(평시 7.36ms → 14.86s).

2차 리팩토링(①접수-매칭 분리 ②홀드 배칭 ③취소 우선 ④거절 품질)은 정확히 이
둘을 겨냥했다. ⑤는 그 코드가 **같은 스파이크에서 실제로 고쳤는지**를 실증한다.
①②③④가 전부 미뤄온 수치(접수 p95·취소 실패율·셰딩 분포)를 여기서 처음 보고한다.

## 왜 이 방식을 선택했는가

### same-session A/B (드리프트 완전 제거)

- **A = 2차 이전 tip**: ①의 첫 커밋의 부모(=①②③④를 정확히 뺀 main). 순수하게
  2차의 효과만 격리한다. 정확한 해시는 구현 계획에서 `git log`로 확정.
- **B = 현 main**(④까지, `daf9d3e`).
- 같은 세션·같은 VM에 A→B 순차 배포. 17~21번이 확립하고 18번이 +24% 드리프트로
  입증한 same-session 원칙 계승 — 세션/VM 차이를 0으로 만들어 "같은 조건에서
  옛 코드 43.9% 실패 / 새 코드 0%"를 airtight하게 만든다.
- categorical 비교(취소 43.9%→0, 14.8s→ms)만으로도 드리프트를 압도하지만,
  이력서 맥락에서 same-session A/B의 서사가 더 강력하므로 A를 재측정한다.

### 환경 (22번과 동일 — e2-highcpu-4)

서버 `e2-highcpu-4`, DB `e2-highcpu-4`, load-gen `e2-standard-8`, k6, 서울 리전.
세션 내내 VM stop/start 없음. 22번 수치와 직접 비교 가능한 유일한 구성.
(참고: 프로덕션 고정 목표가 e2-medium/e2-small이라면 그 하드웨어 재측정은 별도
사이클 — 범위 밖.)

### 부하 프로파일 (단일 k6 스크립트, 바이너리 무관)

`300 웜(1분) → 5,000 램프 → 5,000 hold 3분(목표) → ~10,000 버스트 30~60초(초과)
→ 300 급락 → 회복 2분`. 22번 `order-spike-single-symbol.js`(BTC 단일, 취소·시장가
포함) 계승 + 목표 5,000·초과 버스트·503/Retry-After 계측 추가.

- **바이너리 무관**: 스크립트는 요청을 쏘고 결과만 기록(200/503/500·지연·Retry-After
  헤더 존재). A(옛 코드)엔 입장 게이트·Retry-After가 없어 초과 구간에서 매달림/취소
  500이 나오고, B(신)는 빠른 거절(503+Retry-After)이 나온다 — 같은 스크립트가 A는
  "고장", B는 "우아한 셰딩"으로 기록한다.
- **초과 버스트의 목적**: ②가 올린 천장이 5,000을 넘으면 5,000 hold는 셰딩 0으로
  통과하므로, 빠른 거절(④)을 관측하려면 천장을 확실히 넘는 부하가 필요하다.
  천장 위치 자체가 ⑤의 측정 대상이다.

## 측정 시퀀스

```
배포 A → 리셋 → A 런 → 드레인 → 정합성 5검사 → 수집
       → 리셋 → 배포 B → 리셋 → B 런 → 드레인 → 정합성 5검사 → 수집
```

리셋 = 9테이블 `TRUNCATE ... RESTART IDENTITY CASCADE` + backend 재시작 +
`bootstrap loaded=0` 확인 (19~22번과 동일). 배포 = `git archive` → `scp` →
`sudo tar` 추출 → 이전 bench `.env` 복사 → `docker compose ... up -d --build backend`.

## 수집 항목 (22번 계승 + ④ 신규)

- k6 custom 메트릭: 취소 결과(성공/이미체결/실패)·시장가·**503 카운트 + Retry-After
  헤더 존재 검증**.
- 도커 로그 `--since/--until` 스테이지별 `POST /orders` 지연 재구성(k6는 전체 집계만
  주므로 스테이지 분해 필요) — 평시/5,000 hold/초과 버스트/회복 각 구간.
- 채널 게이지(execution/order) 15초 간격 샘플(hold·버스트 포함).
- **`orders_admission_rejected_total{stage}` 런 종료 시 읽기(B 전용)** — 셰딩 분포
  (engine_gate/engine_handoff/coordinator).
- CPU(서버+DB) hold 중반·버스트 중반 실측.
- pprof 30초 best-effort(B) — 아래 함정 참조.
- backend `RestartCount`(`docker inspect`).
- 정합성 5검사(드레인 확인 후): 리컨실리에이션 1(원장-지갑)·2(자산 총량 보존)·
  `failed_settlements`·`failed_market_completions`·시장가 PENDING/PARTIAL 잔존·
  outbox PENDING 잔존·`settlement_batch_fallbacks_total`.

## 성공 기준 (판정)

| # | 항목 | A(옛) 예상 | B(신) 합격 기준 | 검증 |
|---|---|---|---|---|
| 1 | 취소 실패율 | 높음(≥43.9%) | **0** | ③ |
| 2 | 접수 p95 @5,000 hold | 초 단위 | **ms 단위**(목표 한 자릿수, 허용 low-double-digit) | ①② |
| 3 | 초과 버스트 응답 | 매달림/타임아웃 | **빠른 거절**: 503+Retry-After, 지연 바운디드(초 단위 매달림 0) | ①④ |
| 4 | `admission_rejected{stage}` | (없음) | 초과 구간에서 증분(셰딩 분포 기록) | ④ |
| 5 | 정합성 5검사 | 0 | **0** (비협상) | ②③④ |
| 6 | `RestartCount` | 0 | **0** | — |
| 7 | 회복 | 느림 | 급락 후 초 단위 복귀 | ①③ |

**판정 2의 바**: "한 자릿수 ms"를 목표로 하되, 접수 경로(persist+배치홀드+바운디드
핸드오프)의 실제 지연을 측정값 그대로 보고한다. 초 단위(14.8s)에서 ms 단위로의
자릿수 개선이 본질이고, 정확한 값은 측정이 결정한다.

**셰딩 정직성**: 5,000 hold에서 셰딩이 0이든 일부든 그대로 보고한다. 성공은
"항상 바운디드 응답 + 취소 0 + 정합성 0"이지 "셰딩 0"이 아니다. 5,000에서 셰딩이
없으면 천장이 5,000 위, 일부 있으면 천장이 5,000 아래지만 응답은 여전히 바운디드
(둘 다 가용성 정의상 합격) — 실제 천장 위치를 보고한다.

## 함정·규율

- **도구 디스패치 지연**(22번 교훈 #4): gcloud ssh 스냅샷을 harness
  `run_in_background`로 디스패치하면 9~18분 불규칙 지연 재현. 로컬 `& disown`
  백그라운딩으로 회피. 세션 착수 전 재현/회피를 먼저 검증한다. 이 지연은 k6·정합성
  검사(VM에서 독립 완주)엔 무영향 — 영향은 원격 스냅샷 타이밍뿐.
- `DEV_TOOLS_TOKEN`은 트랜스크립트에 절대 노출 금지. 측정 후 VM stop.
- **정합성 0은 비협상** — 위반이 나오면 예단 없이 그대로 기록(22번의 취소 43.9%
  발견처럼, 사전에 없던 위반도 발견 즉시 기록하는 관례).
- **⑤가 수치를 처음으로 주장하는 곳** — ①②③④는 전부 "실증은 ⑤"로 미뤘다.

## 검토한 대안

- **B만 측정, 22번 A와 categorical 비교**: 취소 43.9%→0·14.8s→ms는 자릿수 차이라
  드리프트로 설명 불가 → cross-session이어도 결론 불변. GCP 시간 절반. 다만
  same-session A/B의 이력서 서사가 더 강력해 기각(A 재측정 채택).
- **22번 정확 재현(3000 hold)**: 직접 A/B지만 목표 5,000을 못 침. 5,000 A/B가
  더 높은 부하에서 더 좋음을 a fortiori로 증명하므로 기각.
- **production e2-medium/e2-small 측정**: 22번 수치와 직접 비교 불가. 별도 사이클로
  분리(범위 밖).

## 결과물 (outcome)

- `docs/benchmarks/23-<날짜>-availability-spike-ab.md` — A/B 판정표 + 원본 출력
  ([벤치마크 추적 관례](../../benchmarks/) 계승).
- 이 측정으로 2차 리팩토링(가용성 100%)이 코드+실증까지 완결. 취소 0·접수 ms·
  빠른 거절·정합성 0이 확인되면 ①②③④의 미뤄둔 수치가 전부 채워진다.
