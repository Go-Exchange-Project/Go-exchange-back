# ② 자금 홀드 그룹커밋 설계 (2차 리팩토링 · 가용성)

- **날짜**: 2026-07-19
- **상태**: 설계 검토 중
- **로드맵**: [2차 리팩토링 ②](../../refactor/README.md) — 가용성 100%의 둘째 조각(천장 올리기)
- **근거**: [21번](../../benchmarks/21-2026-07-17-b3-sharding-remeasurement.md)·[22번](../../benchmarks/22-2026-07-18-outbox-ab-and-spike.md) — DB CPU가 병목(idle 4.5%), pprof에서 `holdOrderAssets`가 CreateOrder 경로의 26%. 주문 생성 시 자금 홀드의 DB 왕복이 처리량 천장을 결정한다.

## 왜 필요한가

①(접수-매칭 분리)은 응답을 엔진 매칭에서 떼어냈지만, 접수 자체는 여전히 동기 DB
트랜잭션(주문 영속화 + 자금 홀드)을 거친다. 이 홀드가 **주문마다 DB 왕복 여러 회**
(지갑 FOR UPDATE → 잔고 확인 → available→locked UPDATE → 원장 INSERT)를 쓰고,
21·22번에서 이 경로가 DB CPU를 포화시켜 **처리량 천장 ~1,500건/s**를 만든다.

목표 용량(동시 ~5,000 제출자 ≈ 3,000건/s)은 이 천장을 넘는다. B-4(정산 그룹커밋)가
정산 경로에서 DB 왕복을 배치로 N:1 상각해 효과를 실증했듯, **같은 기법을 주문 생성
홀드에 적용**해 천장을 목표 위로 올린다. 천장이 오르면 ①의 입장 게이트·바운디드 거절이
목표 부하에서 거의 발동하지 않게 되어, 가용성 하드 보장이 실질이 된다.

**B-4보다 까다로운 두 지점**(이 설계가 푸는 것):
1. 정산 이벤트는 이미 단일 워커 큐로 흘러 자연스러운 배칭 지점이 있었다. 주문 홀드는
   **N개 동시 HTTP goroutine에서 각자 발생** — 모으는 계층이 새로 필요하다.
2. B-4의 trade 실패는 드물어 "배치 전체 롤백 → 단건 폴백"으로 충분했다. 주문 홀드의
   **잔고 부족은 흔한 정상 케이스** — 배치 안에서 개별 골라내야 한다.

## 설계

### 섹션 1 — 아키텍처 & 흐름

**새 컴포넌트: 홀드 코디네이터**(단일 goroutine, OutboxWriter 계열의 stateful 컴포넌트).
`CreateOrder`의 DB 트랜잭션 단계를 코디네이터 제출로 대체한다. ①의 게이트·바운디드
핸드오프는 그대로 바깥에 둔다:

```
CreateOrder (①+② 결합):
  1. [①] 입장 게이트: IsIntakeAdmissible false → 503
  2. [②] 코디네이터 있으면 → {order, resultCh} 바운디드 제출 → resultCh 대기
          코디네이터 없으면(단위 테스트·미배선) → 기존 단건 persist+hold 직접 실행
  3. 결과: 성공(order+id) / 잔고부족(409) / 시스템거절(503)
  4. [①] 성공 시 → 바운디드 엔진 핸드오프(TrySubmitOrder) → 200
```

코디네이터 goroutine:
```
for {
  first := <-input                               // 블로킹 수신
  batch := collectBatch(first, input, maxBatch)  // ~5ms 창 논블로킹 drain (적응형)
  HoldBatch(batch)                               // 한 트랜잭션 (섹션 2)
  각 요청의 resultCh에 개별 결과 시그널
}
```

- **적응형**: 저부하 batch=1(추가 지연 ~0), 고부하에서만 배치가 커져 DB 왕복 N:1 상각 — B-4·outbox writer와 동일.
- **영속화+홀드를 한 트랜잭션에**(현재도 한 txn). 배치 INSERT(RETURNING id)로 주문 ID를 받아 그 ID로 홀드 원장을 기록. 둘을 쪼개면 왕복 절감이 무의미하므로 함께.
- **①과의 층위**: ②는 "주문을 원장에 안착(persist+hold)"만, 엔진 핸드오프는 ①. 깔끔히 분리.

### 섹션 2 — `HoldBatch` 트랜잭션 (정합성 코어)

**최상위 불변식: 등가성.** 배치 처리의 최종 상태(지갑·원장·주문)는 같은 순서로 단건
처리 N회를 실행한 결과와 필드 단위로 정확히 같아야 한다(B-4의 등가성 불변식과 동일).

한 `s.DB.Transaction` 안에서, 순서 고정:
```
1. 지갑 키 수집: 각 주문의 홀드 대상(매수=유저 KRW, 매도=유저 코인) dedup.
2. 지갑 일괄 락: FindByKeys → 고유 지갑 ID 오름차순 FOR UPDATE(LockByIDs).
   정산 lockSettlementWallets와 같은 전역 락 순서 → 정산·취소 txn과 데드락 불성립.
3. 순차 fold-검증(제출 순서, 전부 메모리):
   - 주문의 필요 홀드를 그 지갑의 fold된 잔고에 검증.
   - 충분 → 홀드 적용(available→locked를 메모리 사본에 fold), 통과 분류.
   - 부족 / 지갑 없음 → 실패 분류(ConflictError "insufficient", 기존과 동일).
4. 통과 주문만 배치 INSERT(RETURNING id) — 실패 주문은 행조차 안 남김.
5. 변경된 지갑 일괄 UPDATE(fold된 최종 잔고, BatchUpdateBalances).
6. 통과 주문의 OrderHold 원장 일괄 INSERT(CreateMany, 4의 새 id 참조).
7. 커밋.
8. 시그널: 통과 → 성공(order+id) / 실패 → ConflictError(409).
```

**개별 실패 격리 = 핵심.** 한 배치가 성공(홀드 적용)과 실패(잔고 부족)를 섞어 커밋한다.
잔고 부족은 흔한 정상 케이스이므로 통과분만 INSERT/홀드하고 실패분은 **행을 안 남긴다**
— 현재 단건 동작(잔고 부족 시 롤백 → 주문 미생성)과 정확히 일치. 트랜잭션은
all-or-nothing이지만 "통과분만 쓰기"로 실패분이 통과분을 죽이지 않는다.

**같은 유저의 배치 내 다중 주문**: 3번 fold가 제출 순서로 잔고를 순차 차감 → 후행 주문은
선행 차감을 본 잔고로 검증. 단건 순차 실행과 동일 관점 → 등가성 + overspend 방지(같은
KRW로 두 주문 시 둘째가 초과면 정확히 걸러짐).

**연산 순서 주의**: 지갑 락(2)이 검증(3)보다 먼저 — stale 잔고 방지(B-4와 동일 규율).
원장(6)은 주문 INSERT(4) 뒤 — ReferenceID가 새 order.ID.

### 섹션 3 — 실패 처리

- **개별 잔고 부족**(흔함): 섹션 2에서 배치 내 격리(409 개별 시그널). 배치 안 죽임.
- **배치 전체 DB 실패**(드묾 — DB 이상·직렬화 충돌): **단건 폴백**(B-4 동형). 배치 각
  요청을 기존 단건 경로(`holdOrderDirect`)로 개별 재처리 후 시그널. 큰 txn 하나가 32개를
  죽이는 것보다 N개 개별 txn 격리가 안전(직렬화 충돌은 txn 작을수록 감소). 정합성의
  진실은 계속 단건 경로에.
- **코디네이터 입력 오버플로**(목표 초과): **바운디드 제출 → 503**. ①의 엔진-유입
  게이트와 별개의 두 번째 입장 지점(다른 큐, 방어 심층화). 정책 통합(임계값·Retry-After)은
  ④. **결과 대기엔 타임아웃 없음** — 대기를 끊는데 코디네이터가 커밋하면 고아 주문
  발생. 대신 입력(제출)을 바운디드로 막는다. 제출된 요청은 코디네이터가 유한 시간에
  커밋-or-실패로 항상 시그널(배치가 유한) → 대기는 항상 유한, 고아 위험 0.
- **graceful shutdown**: 코디네이터가 종료 도미노 합류 — HTTP 종료 → 코디네이터 입력
  drain(in-flight 배치 커밋·시그널 후 close) → 엔진 stop → outbox → 정산. OutboxWriter의
  "입력 close → 잔여 flush → 반환" 패턴. 종료 중 제출은 에러 시그널(유실 0).

### 섹션 4 — 결합 · 설정 · 메트릭

- **단건 경로 = 3역 겸용**: 코디네이터 미배선 경로 + 배치 실패 폴백 + 정합성 진실.
  기존 `holdOrderAssets` txn 로직을 `holdOrderDirect(order)`로 추출해 셋이 공유(중복 금지).
- **설정**: `GOEXCHANGE_HOLD_BATCH_SIZE`(기본 ~64) + 수집 창 ~5ms(outbox FlushInterval과
  동일). 23번에서 튜닝.
- **메트릭**(23번 측정 수단, B-4 `settlement_batch_size`와 동형): `hold_batch_size`
  히스토그램(1에 몰리면 배칭 미발동), `hold_batch_fallbacks_total`(0이어야 정상),
  코디네이터 입력 채널 길이 게이지.
- **단일 코디네이터 병목 여부 = 측정으로**: 코디네이터는 단일 goroutine이 배치 txn을
  직렬 실행(그룹커밋 지점). DB가 무거운 일을 하므로 N:1 이득이 정상. 23번에서 코디네이터가
  병목이면 **샤딩 코디네이터(유저 해시별)로 승격** — 측정 기반 에스컬레이션.

## 검토한 대안

- **코디네이터 없이 홀드당 비용만 축소**(`UPDATE ... RETURNING` 원자화 등): 새 컴포넌트
  0이지만 이득이 상수 배수(왕복 3→2)라 pprof의 왕복/fsync 바운드에서 2배 천장 목표에 못
  닿을 위험. B-4가 N:1 배칭이 큰 지렛대임을 실증. 기각(측정 없이는).
- **샤딩 코디네이터 선착수**: 락 경합 분산하나 측정 없는 조기 최적화. 단일 코디네이터가
  병목으로 드러날 때 승격(위 에스컬레이션). 기각(지금은).
- **홀드 비동기화**(접수 즉시 200, 홀드는 나중): overspend 방지가 깨짐(같은 잔고로 두
  주문 접수 가능). 자금 홀드는 동기여야 한다. 기각.

## 검증 계획

1. **등가성**(핵심): 같은 주문 시퀀스를 코디네이터/단건 양 경로로 처리 → 지갑·원장·주문
   필드 단위 동일. 같은 유저 다중 주문(fold), 통과/실패 혼합 포함.
2. 개별 격리: 잔고 부족 섞인 배치 → 실패분 409·행 0, 통과분 홀드 성공.
3. 단건 폴백: 배치 txn 실패 주입 → 각 요청 단건 처리.
4. 바운디드 제출: 입력 만석 → 503.
5. 데드락 안전 + `-race`: 동시 CreateOrder + 정산/취소가 같은 지갑 경합.
6. graceful shutdown drain: in-flight 홀드 커밋, 유실 0.
7. 리컨실리에이션(A-6) 위반 0.
8. 기존 CreateOrder·정산·부트스트랩·outbox 통합 테스트 무수정 그린.
9. **성능 실증은 ⑤(23번)** — ②는 처리량 수치를 주장하지 않는다. 성공 기준은
   "등가성 + 격리 + 폴백 + 회귀 그린"까지.

## 재사용할 기존 프리미티브 (B-4에서 구축)

- `WalletRepository.FindByKeys([]WalletKey)`, `LockByIDs(ids)`, `BatchUpdateBalances`.
- `LedgerRepository.CreateMany`.
- 홀드 산술: `applyBuyOrderHold`/`applySellOrderHold`(fold용, 메모리 사본에 적용).
- 신규: 주문 배치 INSERT(`OrderRepository.CreateOrders([]*Order)` 또는 GORM slice create).

## 범위 밖

- ③(취소 우선 경로), ④(입장 정책 정교화 — 두 입장 지점 통합·Retry-After), ⑤(23번 실증).
- 홀드의 인메모리화(측정 후 조건부, 별도 대공사).
- 샤딩 코디네이터(측정 후 조건부 승격).
