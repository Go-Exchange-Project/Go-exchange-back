# B-4: 정산 그룹커밋 설계

- **날짜**: 2026-07-15
- **상태**: 설계 검토 중
- **전제**: 심볼 파티셔닝 정산 워커(A-3에서 구축 — 같은 심볼은 항상 같은 워커가 FIFO 처리)

## 왜 필요한가

현재 trade 1건 = 정산 트랜잭션 1개다. 트랜잭션 하나가 DB 왕복 약 12회를 쓴다:

| 단계 | 왕복 |
|---|---|
| 멱등성 키 조회 | 1 |
| trade INSERT (ON CONFLICT) | 1 |
| 주문 락 (buy, sell 각각 FOR UPDATE) | 2 |
| 지갑 ID 확보 (참가자 지갑 4개) | 4 |
| 지갑 락 (LockByIDs) | 1 |
| 주문 UPDATE × 2 | 2 |
| 지갑 BatchUpdateBalances | 1 |
| 원장 CreateMany | 1 |
| outbox PROCESSED 마킹 (트랜잭션 흡수, 18번) | 1 |
| + COMMIT(fsync) | 1 |

서버-DB가 별도 VM(VPC)이라 왕복이 지배 비용이다. 이는 추측이 아니라 실측이다:
[18번 벤치마크](../../benchmarks/18-2026-07-13-outbox-markprocessed-absorption.md)에서
**왕복 1회(MarkProcessed)를 트랜잭션에 흡수한 것만으로 처리량 +20.6%**.
trade N건을 한 트랜잭션으로 묶으면 왕복이 12×N → 배치당 상수 ~9회가 된다.

## 지켜야 할 불변식

1. **정합성 등가**: 배치 정산의 최종 상태(지갑·원장·주문·trade)는 같은 순서로
   단건 정산 N회를 실행한 결과와 정확히 같아야 한다.
2. **멱등성**: idempotency_key 중복은 no-op + payload 일치 검증(기존과 동일).
3. **데드락 불성립**: 지갑 락은 전역 ID 오름차순(기존 2단계 락 패턴의 확장).
4. **순서 보존**: 같은 심볼의 trade들 → MarketOrderDone 순서가 깨지면 안 된다
   (Done이 trade 정산을 앞지르면 시장가 완료가 유실된다 — A-3에서 확립).
5. **크래시 내구성**: 배치 커밋 전 크래시 → outbox PENDING → 부팅 리플레이가
   재처리(A-3 보장 그대로. 배치는 커밋 단위만 바꾸지 write-ahead 불변식은 그대로).
6. **실패 격리**: 배치 안의 불량 trade 1건이 나머지 N−1건을 죽이면 안 된다.

## 설계

### 1. 배치 수집 (워커 루프)

- 첫 이벤트는 블로킹 수신, 그 뒤 큐에 **이미 쌓여 있는** 이벤트를 논블로킹으로
  최대 32건까지 drain.
- **티머 없음.** OutboxWriter와 달리 여기 이벤트는 이미 outbox에 커밋된 뒤라
  모으려고 기다릴 이유가 없다. 부하가 낮으면 배치=1(단건과 동일, 추가 지연 0),
  백로그가 있을 때만 배치가 자연히 커지는 적응형이다.
- **MarketOrderDone은 배치 경계.** trade만 묶고, drain 중 Done을 만나면 거기서
  끊는다. 앞의 trade 배치를 먼저 정산·커밋한 뒤 Done을 기존 단건 경로로 처리
  → 불변식 4 보존.

### 2. SettleTradeBatch 트랜잭션 (배치당 왕복 ~9회 상수)

한 트랜잭션 안에서:

1. **중복 검사**: `SELECT ... WHERE idempotency_key IN (...)` 1회. 중복은
   payload 일치 검증 후 정산에서 제외(outbox 마킹만 대상에 남김).
2. **신규 trade 배치 INSERT** (ON CONFLICT DO NOTHING) 1회.
   RowsAffected < 신규 건수면 1과 INSERT 사이에 경쟁자가 있었다는 뜻 —
   배치를 포기하고 단건 폴백으로 넘긴다(멱등 경로가 건별로 처리).
3. **주문 락**: 배치 내 고유 주문 ID를 모아 **ID 오름차순** `FOR UPDATE` 1회.
4. **지갑 확보**: 참가자 (user, symbol) 조합 일괄 SELECT 1회 + 없는 지갑 배치
   생성 ≤1회.
5. **지갑 락**: LockByIDs(ID 오름차순) 1회 — 기존 단건 경로의 2단계 락 패턴을
   배치 스코프로 확장한 것.
6. **산술은 메모리에서**: trade를 큐 순서대로, 락으로 읽어온 주문·지갑의
   메모리 사본에 기존 헬퍼(applyTradeFill, settleBuyerKRW,
   creditBuyerCoinWithAcquisitionCost, settleSellerCoin, creditAvailable)를
   그대로 적용한다. 같은 주문·지갑이 배치 내 여러 trade에 등장해도(대형 테이커
   1건 = trade 여러 건) 읽기 1회·쓰기 1회. 검증(상태·side·심볼 일치)도 trade마다
   기존 함수로 수행하되, 배치 내 선행 trade가 만든 중간 상태를 본다 — 단건
   순차 실행과 동일한 관점.
7. **배치 쓰기**: 주문 배치 UPDATE(CASE WHEN, BatchUpdateBalances와 같은 패턴)
   1회, 지갑 BatchUpdateBalances 1회, 원장 CreateMany(4N행) 1회, outbox
   `UPDATE ... WHERE id IN (...)` 마킹 1회.
8. COMMIT 1회.

**단건 SettleTrade는 그대로 유지한다.** 리플레이·재시도 워커·폴백이 계속 쓰는
정합성 로직의 진실이고, 배치는 그 위의 성능 레이어다. 검증·산술 함수는 두 경로가
공유한다(중복 구현 금지 — 로직이 갈라지면 등가성 불변식이 시간이 지나며 깨진다).

### 3. 실패 격리: 단건 폴백

배치 트랜잭션이 어떤 이유로든 실패하면 → 전체 롤백(원자적, 자금 무변동) →
같은 배치를 **기존 단건 경로로 1건씩 재처리**. 불량 trade만 FailedSettlement로
빠지고 나머지는 정상 정산된다.

- 엔진이 만든 trade는 거의 항상 유효하므로 폴백은 예외 경로다 — 상시 비용 아님.
- 폴백의 락 순서는 기존 단건 경로 그대로(이미 안전 증명됨).
- 이 구조 덕에 배치 경로는 "실패하면 통째로 버려도 되는" 최적화로 남는다.

### 4. 데드락 분석

- **지갑**: 같은 유저가 BTC·ETH를 동시에 거래하면 두 워커의 배치가 같은 KRW
  지갑을 다툰다. 두 배치 모두 자기 배치 전체의 지갑을 ID 오름차순으로 한 번에
  잠그므로 AB-BA 사이클이 성립하지 않는다(단건 경로와 같은 논증, 스코프만 확장).
- **주문**: 심볼당 단일 워커라 정산끼리는 주문 락 경쟁이 없다. 취소·시장가 완료
  경로(단건 락)와의 경쟁은 배치도 ID 오름차순 일괄 락이므로 순서 일관.
- **폴백**: 단건 경로 그대로.

### 트레이드오프

- **락 보유 시간 증가**: 배치 32건이면 관련 지갑들이 트랜잭션 내내 잠긴다 —
  잔고 조회·출금과 경쟁. 상한 32로 캡하고 실측 후 조정한다.
- **배치 실패 시 이중 작업**(롤백 + 단건 재처리): 예외 경로라 감수. 대신 성공
  경로가 항상 빠르다.
- **코드 복잡도**: 정산이 두 경로가 된다. 등가성 테스트(아래)로 두 경로의 결과
  일치를 회귀 검증해 갈라짐을 막는다.

## 검증 계획

1. **등가성**: 같은 trade 시퀀스를 배치/단건 양 경로로 정산 → 지갑·원장·주문
   최종 상태 동일. 같은 주문이 여러 trade에 걸치는 케이스, 같은 유저가 배치 내
   매수자이자 매도자인 케이스 포함.
2. **멱등성**: 배치 내 일부 중복 / 전체 중복 배치 재정산 → no-op + outbox 마킹.
3. **폴백**: 배치에 불량 trade 주입 → 롤백 → 단건 폴백으로 N−1건 정산 + 1건
   실패 기록.
4. **순서**: trade→Done 시퀀스에서 Done이 배치를 끊고 완료가 유실되지 않는지.
5. **원장 정합**: 리컨실리에이션 검사(원장-지갑 일치, 자산 총량 보존)가 배치
   정산 후에도 위반 0 — A-6이 최종 방어선.
6. `-race` 포함 전체 스위트.
7. **성능**: GCP 일괄 재측정 사이클에서 B-1과 묶어 hold 프로파일 same-session
   A/B("나중에 일괄" 방침).

## 구현 순서

```
1. 배치 수집 루프 + Done 경계        → 검증: 수집 단위 테스트
2. SettleTradeBatch(락·산술·배치 쓰기) → 검증: 등가성·멱등성 테스트
3. 단건 폴백 연결                     → 검증: 불량 주입 테스트
4. main.go 워커 교체 + 메트릭          → 검증: 전체 스위트 + -race
```

## 검토한 대안

- **최소형(한 트랜잭션에 단건 로직 N회 반복)**: 구현은 작지만 왕복이 12×N
  그대로라(COMMIT만 절약) 실측된 병목을 안 건드리고, trade마다 지갑을 따로
  잠그면 전역 ID 순서가 깨져 데드락 가능 — 결국 배치 스코프 락이 필요해
  복잡도 이득도 없다. 기각.
- **비동기 정산 확인(커밋 전 전달)**: write-ahead 불변식 위반. 기각.
