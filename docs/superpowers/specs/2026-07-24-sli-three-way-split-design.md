# 3차 ③ SLI 3분할 설계 (측정 규율)

- **날짜**: 2026-07-24
- **상태**: 설계 검토 중
- **로드맵**: [3차 리팩토링 ③](../../refactor/README.md) — ①(취소 500)·②(처리량)를 정확히 재는 토대
- **근거**: 23번에서 "가용성 ~100% / 업무 성공 ~7% / 취소 성공 ~51%"가 뒤섞여 "9,800 VU까지
  버텼다"가 무슨 뜻인지 혼선이 났다. ①②를 재기 전에 **하나로 뭉뚱그린 가용성을 세 개의
  독립 SLI로 분리**해 k6에 baked-in 한다. 이 단계는 **지표 정의 + 기준선 수집**이 목적 —
  합격 목표치(threshold)는 재측정 후에 정한다.

## 왜 필요한가

현 `_workspace/loadtest/order-spike-availability.js`는 order/market/cancel/503 Counter는
있지만 가용성이 **세 가지 서로 다른 의미로 분리 계산되지 않는다.** 특히 주문 503이
`http.expectedStatuses(200,201,503)`으로 정상 취급되어, `http_req_failed`만으로는 다음을
구분할 수 없다:

- 요청이 제한 시간 안에 응답했는가 (응답 가용성)
- 주문이 실제로 접수됐는가 (업무 성공)
- 취소가 인프라 문제 없이 성공했는가 (취소 성공)

이 셋은 과부하에서 **정반대로 갈린다**: 서버가 안 매달리고 빠르게 503을 내면 응답
가용성은 높지만 업무 성공률은 낮다. 그 차이가 곧 **셰딩의 영향**이다.

## 설계 — k6 `Rate` 3개, 응답 처리 지점에서 인라인 `.add()`

지표명에 **order/cancel 스코프를 명시**해 "전체 HTTP 가용성"으로 오해되지 않게 한다.
분모의 "전체"는 **전체 주문 제출 시도**로 정의한다.

### 1. `sli_order_response_availability`

```
((주문 응답 ∈ {200,201,503}) AND res.timings.duration ≤ RESPONSE_SLO_MS) / 전체 주문 제출 시도
```

주문 접수 여부와 별개로 **서버가 제한 시간 안에 계약된 응답을 반환했는지**. 핵심:
**느린 2xx도 실패** — 24초 뒤 받은 201은 업무 성공엔 포함되나 응답 가용성엔 불포함.
`status 0`(k6 타임아웃/연결실패)·기타 예상 밖 상태도 실패.

### 2. `sli_order_business_success`

```
주문 2xx(200/201) / 전체 주문 제출 시도
```

**실제로 접수된 비율**(= 1 − 셰딩률). 빠르게 반환된 503도 여기선 실패. 응답 가용성과의
차이가 과부하 셰딩의 영향을 드러낸다(예: 가용성 ~100%인데 업무 ~7% = 안 매달렸지만
대부분 접수 못 함).

### 3. `sli_cancel_success`

```
취소 200 / (취소 200 + 취소 인프라 실패)
```

상태 분류:
- `200` → 성공(`.add(true)`)
- `404`/`409` → 이미 체결/오더북에서 사라진 **정상 경쟁 결과** → **분모 제외**(`.add()` 호출 안 함)
- `status 0`·`5xx`·기타 예상 밖 → **인프라 실패**(`.add(false)`)

**500만 인프라 실패로 제한하지 않는다** — k6 타임아웃·연결 실패(status 0)가 분모에서
빠지면 지표가 실제보다 좋아진다(측정 무결성).

### 판정표

주문:

| 응답 | 응답 시간 | 응답 가용성 | 업무 성공률 |
|---|---|---|---|
| 200/201 | ≤ SLO | 성공 | 성공 |
| 200/201 | > SLO | **실패** | 성공 |
| 503 | ≤ SLO | 성공 | **실패** |
| 503 | > SLO | 실패 | 실패 |
| status 0/기타 | 무관 | 실패 | 실패 |

취소:

| 응답 | 처리 |
|---|---|
| 200 | 성공 추가 |
| 404/409 | 집계 제외 |
| status 0·5xx·기타 | 실패 추가 |

### RESPONSE_SLO_MS

```js
const RESPONSE_SLO_MS = parseInt(__ENV.RESPONSE_SLO_MS || '1000', 10);
```

기본 **1000ms**. 근거 — 23번 실측: B 주문 2xx p95 hold ~130ms·버스트 ~228ms, 503 대부분
sub-ms, A 2xx p95 ~24초. 1000ms는 B 버스트 p95의 ~4.4배 여유라 일시적 네트워크 흔들림을
과실패 처리하지 않으면서 초 단위 매달림은 명확히 불합격. **이 값은 "스파이크 벤치마크
응답 경계 기본값"으로 정의**하고, 운영 트래픽 데이터 확보 후 프로덕션 SLO로 승격한다
(지금 프로덕션 99.9% SLO로 선언하지 않는다).

### 타이밍

응답 가용성 시간은 `res.timings.duration`(서버 응답 시간)을 쓴다. 503 이후 Retry-After를
존중하는 `sleep()`은 **클라이언트 백오프이지 서버 지연이 아니므로 포함하지 않는다**
(res 수신 후 sleep이라 자연히 분리됨 — 분류는 sleep 전에 수행).

## 유지 / 하지 않는 것

- **기존 Counter·`http.expectedStatuses` 유지** — 별도 이유 없으면 안 바꾼다(Rate는 추가).
- **Rate threshold 안 건다** — 이 단계는 정의+기준선 수집. 합격 목표는 재측정 후.
- **서버 메트릭/Grafana 안 만듦** — 업무·취소 성공률은 기존 `HTTPRequestsTotal{path,method,status}`
  에서 PromQL로 유도 가능(필요 시 문서에 예시만). 응답 가용성은 클라 지연이 홈이라 k6가 적합.
  프로덕션 SLI 대시보드는 YAGNI(후속).
- **whole-run SLI로 시작** — 23번 볼륨상 hold+버스트가 지배해 "스파이크 하 세 지표"를 그대로
  보여줌. 스테이지별 태깅은 whole-run이 판단을 흐린다는 증거가 생기면(도커 로그 재구성은 유지).

## 검토한 대안

- **서버측 SLI 메트릭 신설**: 업무·취소는 기존 HTTP 상태 메트릭에서 유도 가능, 응답 가용성만
  애매 → 신설은 중복. 기각(PromQL 예시로 대체).
- **취소 실패를 500으로만 집계**: status 0/타임아웃이 분모에서 빠져 지표 과대평가. 기각.
- **스테이지별 태깅 즉시 도입**: whole-run이 목적(혼선 해소)엔 충분. 증거 나오면 후속.

## 검증 계획

- **분류 로직 순수 함수 추출 + 러너블 셀프체크**: `classifyOrderResponse(status, durationMs)`,
  `classifyCancelResponse(status)`를 순수 함수로 빼고, 위 판정표 각 행을 assert로 덮는
  최소 셀프체크(프레임워크 없이). 이게 "느린 2xx를 가용으로 잘못 세는" 류의 조용한 버그를 막는다.
- **로컬 k6 스모크**: `STAGEn_*`·소규모 VU 오버라이드로 짧게 돌려 세 SLI가 요약에 출력되고
  값이 sane 한지 확인(전부 성공 런 → 세 지표 ~100%; 503 유발 시 업무 < 가용성).
- **정의 문서화**: SLI 3식 + 판정표 + PromQL 유도 예시(업무·취소)를 벤치마크 방법론에 남겨
  23번 재측정·향후 벤치마크가 통일 보고하게 한다.
