# OutboxWriter 처리율 개선 설계

- **날짜**: 2026-07-17
- **상태**: 설계 검토 중
- **근거**: [21번 벤치마크](../../benchmarks/21-2026-07-17-b3-sharding-remeasurement.md) — flush가 배치 상한 64에 포화(평균 54.4건/flush) 상태로 초당 15.2회(≈66ms/flush) 도는 것을 실측. **처리율 캡 ≈ 54.4 × 15.2 ≈ 827 events/s**. OutboxWriter는 write-ahead 관문(모든 체결이 통과)이므로 이 캡이 곧 전체 파이프라인 캡이고, execution 채널 만석(8192)과 order 채널 만석의 근원이다.

## 지켜야 할 불변식

1. **write-ahead(A-3)**: 정산은 outbox에 커밋된 이벤트만. 커밋 성공 후에만 forward.
2. **백프레셔**: INSERT 실패 무한 재시도, 채널이 차면 매칭 블록 — 의도된 설계, 완화 금지.
3. **심볼 내 순서**: 수집 순서 = INSERT 순서(outbox ID 단조) = forward 순서.
4. **graceful shutdown**: Source close 후 잔여 배치 flush 완료 뒤 반환.

## 설계: 1단계 — 배치 상한 상향 (이번 구현)

**`defaultOutboxBatchSize` 64 → 512, `GOEXCHANGE_OUTBOX_BATCH_SIZE` 환경변수로
조정 가능**(기본 512, `SettlementWorkersFromEnv` 패턴).

- 근거 산술: 66ms/flush가 전부 고정비(왕복+fsync+큐잉)라면 512로 상한 시
  512/0.066 ≈ 7.7k events/s(9.4배). 행당 비용이 지배하더라도 왕복·fsync 횟수가
  1/8이 된다. 실제 이득은 측정으로 확인한다(아래).
- 안전성: GORM 배치 INSERT 파라미터 512행 × ~7컬럼 ≈ 3.6k « Postgres 한도
  65,535. 페이로드 ~350B × 512 ≈ 180KB/INSERT — 문제없는 크기.
- **FlushInterval(5ms)은 유지** — 저부하에서 배치는 티머로 닫히므로 지연 특성
  불변. 포화 시에만 배치가 커진다(B-4·B-1c와 같은 적응 패턴).
- 수집 루프(collectBatch)·재시도·shutdown 로직 무변경 — 바뀌는 것은 상한
  상수의 출처(env)와 메트릭 버킷뿐이다.
- 메트릭: `trade_outbox_flush_batch_size` 버킷이 128에서 끝남 → 256/512/1024
  추가(포화 여부 관측이 이 작업의 판정 수단이므로 필수).

### 상호작용 점검

- **정산 큐(256) < 배치(512)**: 단일 심볼 폭주 시 forward가 워커 큐에 블록될 수
  있다 — 커밋 후이므로 정합성과 무관하고, 의도된 백프레셔다. 큐를 같이 키우지
  않는 이유: 21번의 교훈(병목이 다운스트림일 때 깊은 버퍼는 꼬리 지연만 키운다).
  측정에서 "INSERT가 아니라 forward에서 막힘"이 관측되면 그때 재검토.
- 리플레이·부팅 경로 무변경(읽기 페이지네이션은 별도 상수).

## 라이더: 기본 엔진 샤드 수 NumCPU → 1

21번 백로그 소품. 병목이 다운스트림인 현재, 샤드 N개 = 주문 버퍼 N×1024로
p95만 악화(실측 +57%). 기본을 1로 내리고 `GOEXCHANGE_ENGINE_SHARDS`로 필요 시
확대한다. B-3 코드는 그대로(매칭이 실제 병목이 되는 날의 기반).

## 검토한 대안

- **병렬 writer(심볼 파티션 N개)**: 2단계 후보로 **보류**. 스케치는 준비돼 있다
  — 심볼→writer 고정 배정(first-seen 라운드로빈), writer별 그룹커밋, 같은
  writer가 forward까지 수행하면 심볼 내 outbox ID 단조성과 전달 순서가 보존된다.
  지금 안 하는 이유: 1단계가 10배 싸고, DB CPU가 이미 258%/400이라 병렬화
  이득이 DB에서 막힐 가능성이 높다 — 1단계 재측정에서 **여전히 flush가 새 상한에
  포화하고 execution 채널이 만석이면** 진행한다.
- **파이프라이닝(수집·INSERT 중첩)**: 포화 상태에서 수집은 즉시 완료되므로
  중첩 이득이 0. 기각.
- **내구성 완화(synchronous_commit=off 등)**: 크래시 유실 0 불변식 위반. 기각.

## 검증 계획

1. 단위: env 파싱(기본 512/오버라이드/비정상 폴백), OutboxWriter가 상한까지
   배치를 채우는지(기존 writer 테스트가 BatchSize 파라미터로 이미 커버 — 확장).
2. 기존 outbox writer·리플레이·크래시 계약 테스트 무수정 그린(로직 무변경의 증거).
3. `-race` 포함 전체 스위트.
4. **측정(22번, 다음 사이클)**: 같은 바이너리로 env만 바꾸는 A/B(64 vs 512,
   필요 시 128/256/1024 스위프) — 재빌드 없는 A/B라 배포 변수까지 제거된 가장
   깨끗한 비교. 판정: ① flush 평균이 새 상한 대비 여유(포화 해제) ② execution
   채널 만석 해제 ③ 처리량 상승 폭 ④ 다음 병목(DB CPU 예상) 식별.
