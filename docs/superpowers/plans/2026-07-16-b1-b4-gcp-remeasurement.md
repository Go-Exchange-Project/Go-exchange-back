# B-1/B-4 일괄 GCP 재측정 계획 (19번 벤치마크)

> **For agentic workers:** 이 계획은 코드 구현이 아니라 측정 절차다. 순서대로 실행하고, 각 측정의 원본 출력을 보존한 뒤 19번 벤치마크 문서로 정리한다.

**Goal:** 18번 이후 쌓인 세 변경 — B-1a(스냅샷 코얼레싱+캐시), B-1b(WS 심볼 구독), B-4(정산 그룹커밋) — 의 누적 성능 효과를 같은 세션 A/B로 측정한다. B-1 완료 문서와 B-4 완료 문서의 "GCP 측정은 다음 사이클에 일괄" 잔여 항목을 해소한다.

**측정 대상 커밋:**
- **A (before)**: `bc8c00f` — 18번의 opt(MarkProcessed 흡수), 마지막으로 측정된 코드
- **B (after)**: `f3103c0` — 현 main (B-1a `31f0f80` + B-1b `51ef2e1` + B-4 6커밋 누적)

## 방법론 (17·18번에서 확립, 위반 금지)

- **같은 세션 A/B**: A와 B를 한 VM 세션에서 연속 측정한다. 측정 도중 VM stop/start 금지 — 재시작 시 물리 호스트가 바뀌어 +24% 수준의 세션 교란이 생긴다(18번 실증). 18번 수치와의 직접 비교도 하지 않는다(세션이 다름) — 이번 세션의 A가 유일한 기준선이다.
- **매 측정 전 리셋**: TRUNCATE + backend 재시작(bootstrap loaded=0 확인). 모든 측정이 빈 오더북·cold 풀에서 시작.
- **hold 프로파일**: 1.5분 워밍업 → VU 3000에서 8분 유지 (`~/stress-hold3000.js`, load-gen VM에 있음. 없으면 17번 문서와 git 히스토리에서 복원).
- 실행 커맨드(load-gen VM에서): `k6 run -e BASE_URL=http://10.10.0.3:8080 -e DEV_TOOLS_TOKEN=<토큰> ~/stress-hold3000.js`

## 측정 매트릭스 (5회, 각 ~10분 + 리셋)

| # | 빌드 | WS 클라이언트 | 목적 |
|---|---|---|---|
| 1 | A | 없음 | 기준선 (REST 주문 처리량) |
| 2 | A | N개 legacy full-feed | A에서 WS fanout이 처리량을 깎는 정도 (B-1 이전의 결박) |
| 3 | B | 없음 | B-1a 코얼레싱 + B-4 그룹커밋의 누적 효과 |
| 4 | B | N개 legacy full-feed | 같은 WS 부하에서 결박 해소 확인 (핵심 대조: 2 vs 4) |
| 5 | B | N개 구독(BTC 1심볼) | B-1b 효과 — 클라이언트당 수신 메시지 수가 legacy 대비 감소 |

재배포 최소화를 위해 A로 1·2 → B 배포 후 3·4·5 순서로 진행한다.

**N (WS 클라이언트 수):** 300에서 시작. load-gen이 e2-small이라 REST 3000 VU와 동시 실행 시 메모리 여유를 먼저 확인하고(측정 2 전에 WS 스크립트 단독 스모크), 부족하면 낮춰서 전 측정에 동일하게 적용한다. N은 2·4·5에서 반드시 동일해야 한다.

## WS 부하 스크립트 (신규, k6 `k6/ws`)

- 연결 N개를 측정 시간 동안 유지하며 수신 메시지 수를 Counter로 집계.
- 모드 2개(env로 전환): legacy(연결 후 아무것도 안 보냄), subscribe(연결 직후 `{"action":"subscribe","coin_symbols":["BTC"]}` 전송).
- REST hold와 **별도 k6 프로세스**로 동시 실행(같은 load-gen VM).
- 스크립트는 리포 `scripts/`가 아니라 벤치마크 관례대로 load-gen에 두되, 원문을 19번 문서에 첨부한다.
- **함정**: 서버 WS 핸들러는 Origin 체크가 있다(`ServeWs`). k6가 Origin 없이 연결하면 거부될 수 있으니 서버 env `GOEXCHANGE_WS_ALLOW_MISSING_ORIGIN`(정확한 이름은 `internal/ws` 코드에서 확인) 설정 또는 k6에서 Origin 헤더 지정으로 해결하고, 사용한 방법을 문서에 기록한다.

## B 측정 중 함께 수집 (서버 /metrics 스크레이프)

- `settlement_batch_size` 히스토그램 — B-4 그룹커밋이 실제 발동하는지(분포가 1에 몰려 있으면 발동 안 한 것).
- `settlement_batch_fallbacks_total` — 0이어야 정상.
- 측정 3~5 각각 종료 직후 스냅샷을 떠서 문서에 기록한다.

## 절차

1. **VM 기동**: `goexchange-stress-server` / `goexchange-stress-db` / `goexchange-stress-load-gen` 3대 start (서울 리전, zone은 terraform.tfvars 확인). IAP SSH.
2. **DB 준비**: db VM의 postgres 컨테이너가 자동 기동 안 될 수 있음 — `docker ps` 확인 후 수동 start.
3. **A 배포**: 서버 VM에서 `bc8c00f` 체크아웃·빌드·재배포(기존 절차 — 서버 VM에서 빌드, 17번 방식).
4. 측정 1 → 리셋 → 측정 2 (WS 스모크 먼저).
5. **B 배포**: `f3103c0` 체크아웃·빌드·재배포.
6. 리셋 → 측정 3 → 리셋 → 측정 4 → 리셋 → 측정 5. 각 측정 후 /metrics 스냅샷.
7. **문서화**: `docs/benchmarks/19-2026-07-16-b1-b4-batch-remeasurement.md` — 기존 컨벤션(커밋 해시 / 왜 / 방법론 / 실행한 정확한 커맨드 / 핵심 결과 표 / 원본 출력 / 해석 / 범위 밖). B-1 완료 문서(5번)와 B-4 완료 문서(6번)의 "남은 것" 절도 결과 반영해 갱신.
8. **VM 3대 stop** (비용 — 이전 세션 관례).
9. 커밋(author→reviewer 절차) + 푸시.

## 판정 기준 / 이상 징후 시

- 기대: 3 ≥ 1 (B-1a+B-4 효과), (4−2)의 하락폭 < (2−1)의 하락폭... 정확히는 **4/3 비율이 2/1 비율보다 높아야** WS 결박 해소가 실증된 것. 5는 4 대비 클라이언트당 수신 메시지 수 대폭 감소.
- 3이 1보다 **낮으면**(회귀): 중간 커밋으로 격리 측정한다 — `31f0f80`(B-1a만), `51ef2e1`(B-1b까지)을 같은 세션에서 추가 배포해 어느 변경이 깎았는지 이등분 탐색. 이 경우도 세션을 끝내지 말고 이어서 한다.
- `http_req_failed`가 0이 아니면 원인 파악 전에 다음 측정으로 넘어가지 않는다.
