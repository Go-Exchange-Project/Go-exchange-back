# 다중 심볼 부하 프로파일 + GCP 재측정 계획 (20번 벤치마크)

> **For agentic workers:** 이 계획은 측정 절차다. 순서대로 실행하고, 원본 출력을 보존한 뒤 20번 벤치마크 문서로 정리한다.

**Goal:** 다중 심볼 부하 프로파일을 만들어 세 가지를 한 세션에 측정한다:
1. **B-1c 실측** — 체결 브로드캐스트 배치화가 WS 부하 시 처리량 하락폭(19번에서 −33.5%)을 줄이는가.
2. **B-1b 실측** — 심볼 구독이 클라이언트당 수신 메시지를 심볼 수에 비례해 줄이는가 (19번에서는 단일 심볼 프로파일이라 측정 불가였음).
3. **엔진 병목 판정** — 다중 심볼 부하에서 매칭 엔진(단일 goroutine)이 병목인가. **B-3(심볼 샤딩) 진행 여부를 이 데이터로 결정한다** (16번 교훈: 병목이 아닌 곳을 먼저 최적화하지 않는다).

**측정 대상 커밋:**
- **A (before)**: `f3103c0` — B-1c 직전 (19번 벤치마크의 B와 동일 코드)
- **B (after)**: 현 main — B-1c(체결 브로드캐스트 배치화) 포함

## 방법론 (17·18·19번 계승, 위반 금지)

- **같은 세션 A/B**: 서버·DB VM은 측정 도중 stop/start 금지(+24% 세션 교란, 18번 실증). 19번 수치와의 직접 비교 금지 — 프로파일도 세션도 다르다. 이번 세션의 A가 유일한 기준선.
- **매 측정 전 리셋**: TRUNCATE(19번 문서의 9개 테이블 커맨드) + backend 재시작 + bootstrap loaded=0 확인.
- **hold 프로파일**: 1.5분 워밍업 → VU 3000에서 8분 유지.
- **load-gen은 e2-standard-8 유지**(19번에서 k6 OOM으로 리사이즈, stopped 상태로 보존됨). B는 19번의 861 iter/s급이므로 k6 리소스를 측정 중 감시한다.

## 사전 작업: 부하 스크립트 (리포에 커밋 — 19번에서 load-gen 디스크 유실 재발 방지)

1. **`loadtest/order-submission-multisymbol.js` (신규)** — 기존 hold 프로파일(19번 문서 첨부본) 기반, 심볼 8개로 확장:
   - `SYMBOLS = ['BTC','ETH','XRP','SOL','ADA','DOGE','TRX','DOT']` — 마켓 룰 레지스트리는 미등록 심볼에 기본 정책을 적용하므로 서버 수정 불필요(`internal/service/market_rules_registry.go` 확인됨. 단 "HALT"는 금지 심볼). FIXED_PRICE는 기존 50000000 유지 — 틱 룰은 가격 기반이라 전 심볼 동일 적용.
   - **매수/매도 쌍이 같은 심볼이어야 체결이 난다**: 유저 i의 심볼을 `floor((i-1)/2) % 8`로 배정해 (buyer, seller) 홀짝 쌍이 항상 같은 심볼을 갖게 한다. 심볼당 유저 375쌍.
   - setup의 지갑 충전도 심볼별로(매도자는 자기 심볼 코인, 매수자는 KRW).
2. **`loadtest/ws-load.js` (19번 첨부본을 리포로 승격 + 확장)** — subscribe 모드의 구독 심볼을 env로(기본 BTC 1개).
3. 두 스크립트를 load-gen VM에 복사해 사용.

## 측정 매트릭스 (5회, 전부 다중 심볼 프로파일)

| # | 빌드 | WS | 목적 |
|---|---|---|---|
| 1 | A | 없음 | 다중 심볼 REST 기준선 |
| 2 | A | legacy×300 | B-1c 이전의 WS fanout 하락폭 |
| 3 | B | 없음 | REST 동등성 확인(B-1c는 WS 없으면 효과 미미해야 정상) + **엔진 병목 데이터 수집** |
| 4 | B | legacy×300 | **핵심 대조: (4/3) vs (2/1)** — B-1c가 하락폭을 줄였는가 |
| 5 | B | subscribe×300(BTC 1심볼) | **B-1b**: 클라이언트당 수신 메시지가 4 대비 대폭(이론상 ~8:1) 감소하는가 |

재배포 최소화: A로 1·2 → B 배포 후 3·4·5. WS 클라이언트 수 300은 2·4·5 동일 고정.

## 엔진 병목 판정 데이터 (측정 3 중 수집)

- **/metrics 채널 게이지**: 매칭 엔진 입력 채널 길이(`RegisterMatchingEngineChannelLenGauges` 등록분)를 측정 중 30초 간격 샘플링 — 지속 적체 여부.
- **pprof CPU 프로파일 30초** (hold 구간 중반): `GOEXCHANGE_ENABLE_PPROF` 활성화(4·8번 벤치마크 절차) 후 매칭 루프 goroutine의 CPU 점유 확인.
- **판정 기준**:
  - **B-3 진행**: 매칭 엔진 goroutine이 ~1 vCPU를 포화(4 vCPU 서버에서 프로파일 상 매칭 루프가 코어 하나를 지배)하거나 엔진 입력 채널이 지속적으로 적체.
  - **B-3 보류**: 정산 워커/DB 대기/HTTP가 지배적이고 엔진 채널이 한산 — 이 경우 README에서 B-3를 백로그로 강등하고 사유(이 측정)를 남긴다.

## 판정 기준 (B-1b/B-1c)

- B-1c 실증: **(측정4 ÷ 측정3) > (측정2 ÷ 측정1)** — WS 부하로 인한 상대 하락폭 감소. 보조 지표: 측정 4의 `ws_messages_received`가 측정 2 대비 큰 폭 감소(배치 ~32:1, 단 스냅샷 8심볼×10/s×300은 잔존).
- B-1b 실증: 측정 5의 클라이언트당 수신 메시지 < 측정 4의 1/4 (이론 ~1/8, 전역 ticker 등 잔존분 감안).
- 측정 3 vs 1이 크게 다르면(B-1c는 REST 단독에 거의 중립이어야 함) 원인을 파악하고 기록한다.
- `http_req_failed` > 0이면 원인 파악 전에 다음 측정으로 넘어가지 않는다. 예상 밖 결과도 그대로 기록한다(19번 관례).

## 절차

1. 스크립트 2개 작성·리포 커밋(사전 작업) — 로컬에서 소규모 스모크(유저 16명, VU 8, 1분)로 심볼 8개 전부 체결·정산되는지 확인 후 GCP로.
2. VM 3대 start(`goexchange-stress-server/-db/-load-gen`) → db postgres 컨테이너 확인 → 스크립트 load-gen 복사.
3. A(`f3103c0`) 배포 → 리셋 → 측정 1 → 리셋 → 측정 2.
4. B(main) 배포 → 리셋 → 측정 3(+ 채널 게이지 샘플링 + pprof) → 리셋 → 측정 4 → 리셋 → 측정 5. B 측정마다 `settlement_batch_size`/`fallbacks` 스냅샷도 수집(19번 관례).
5. 문서화: `docs/benchmarks/20-...md`(기존 컨벤션) + 5번·7번 완료 문서의 잔여 항목 갱신 + **README의 B-3(8번) 행을 판정 결과대로 갱신**(진행 확정 또는 백로그 강등).
6. VM 3대 stop → 커밋(author→reviewer) + 푸시 + CI 확인.

## 범위 밖

- 프론트엔드 구독 opt-in(별도 리포), 시장가 경로 벤치마크.
