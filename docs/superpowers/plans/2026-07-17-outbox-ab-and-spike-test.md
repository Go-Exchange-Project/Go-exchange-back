# 22번 벤치마크 계획: outbox 상한 실증 + 급등락 스파이크 내성

> **For agentic workers:** 이 계획은 측정 절차다. 사전 작업(코드 소품 2건 + 스크립트 1개)을 커밋한 뒤 측정을 수행하고, 원본 출력을 보존해 22번 문서로 정리한다.

**Goal:** 두 가지를 한 세션에 측정한다.
- **1부 — outbox 상한 실증(9번 후속)**: `GOEXCHANGE_OUTBOX_BATCH_SIZE` 64 vs 512를 **같은 바이너리** env 스위프로 A/B. 판정 4개: ① flush 포화 해제(평균 « 상한) ② execution 채널 만석 해제 ③ 처리량 상승 폭 ④ 다음 병목 식별.
- **2부 — 급등락 스파이크 내성(신규)**: 단일 심볼 집중 + 시장가·취소 혼합 + 급격한 램프의 스파이크 프로파일에서 ① 죽지 않고 ② 에러 없이 ③ 정합성 위반 0으로 버티며 ④ 스파이크 후 회복되는지. 기존 벤치마크가 커버하지 않던 실전 시나리오(지정가 전용·완만한 램프의 한계)를 처음 측정한다.

## 방법론 (17~21번 계승, 위반 금지)

- 같은 세션(서버·DB VM stop/start 금지), 매 측정 전 리셋(TRUNCATE + backend 재시작 + loaded=0), 21번 이전 수치와 직접 비교 금지.
- **1부는 same-binary A/B**: 배포는 1회(현 main), 측정 간에는 서버 `.env`의 `GOEXCHANGE_OUTBOX_BATCH_SIZE`만 바꾸고 backend 재시작 — 역대 가장 깨끗한 A/B(빌드·배포 변수 0).
- 2부는 최적 구성(batch=512, 샤드 기본 1)에서 1회 — 내성 측정이지 A/B가 아니다.

## 사전 작업 (리포 커밋 — 측정 전에)

1. **compose env 전달 라인**: `docker-compose.stress.yml` environment에 `GOEXCHANGE_OUTBOX_BATCH_SIZE: ${GOEXCHANGE_OUTBOX_BATCH_SIZE:-}` 추가. **B-3 측정 때 이 누락으로 shards=4 사고가 났던 함정과 동일** — 반드시 먼저 막는다.
2. **기동 로그 1줄**: `cmd/main.go`에 `log.Printf("outbox writer batch size=%d", ...)` — env 적용을 기동 로그로 확인할 수단(shards 로그가 B-3 사고를 잡았던 관례). 이 로그 확인 없이는 측정을 시작하지 않는다.
3. **스파이크 스크립트** `loadtest/order-spike-single-symbol.js` (신규, 아래 사양). 로컬 스모크(유저 20명, 1분 축소판) 필수 통과 후 GCP로.

### 스파이크 스크립트 사양

- **BTC 단일 심볼**, 유저 3,000명(기존 setup 로직 재사용 — 매수자 KRW·매도자 BTC 충전).
- **역할 믹스**: 메이커 60%(지정가를 기준가 위/아래 1~5틱에 rest시켜 오더북 유동성 축적 — 기존 프로파일의 단일 고정가는 즉시 체결돼 오더북이 안 쌓이므로 반드시 가격 분산), 테이커 40%(절반 시장가 — **MARKET BUY는 price=0·amount=0·quote_amount>0 제약 주의** —, 절반 crossing 지정가).
- **취소 폭주**: 메이커는 제출 후 1~3초 뒤 30% 확률로 자기 주문 취소. 취소 응답은 200(성공)/404·409(이미 체결 — 정상)를 구분 집계하고 후자는 실패로 세지 않는다.
- **스테이지(급등락 모사)**: 평시 2m@VU300 → **30초 만에 VU3000 급등** → 3m@3000 유지 → 30초 급락 → 회복 관찰 2m@300.
- 커스텀 카운터: 주문 성공/실패, 시장가 성공/실패, 취소 성공/이미체결/실패.

## 측정 매트릭스

| # | 프로파일 | env | 목적 |
|---|---|---|---|
| 1 | 다중 심볼 hold(기존) | `OUTBOX_BATCH_SIZE=64` | 1부 기준선(A) — 21번 조건 재현 |
| 2 | 다중 심볼 hold(기존) | `OUTBOX_BATCH_SIZE=512` | 1부 B + 수집(flush 분포·duration, execution 채널 게이지 30초 샘플, DB CPU, pprof) |
| 3(조건부) | 다중 심볼 hold | `=1024` 또는 `=256` | 2의 이득이 애매(±5%)하거나 512도 포화면 스위프 1점 추가 |
| 4 | **스파이크(신규)** | `=512` | 2부 — 내성·회복·정합성 |

측정 4 중 수집: 채널 게이지·정산 큐 30초 샘플(스파이크 구간 포함), 서버·DB CPU, backend 크래시/재시작 여부(`docker ps` 재시작 카운트).

## 측정 4 종료 후 정합성 검사 (필수 — 이게 2부의 핵심 판정)

정산 드레인 대기(execution 채널·정산 큐 0 확인) 후:

1. **리컨실리에이션 검사 SQL**(`docs/gcp-db-restore-runbook.md`의 검사 쿼리 재사용): 원장-지갑 일치·자산 총량 보존 위반 **0건**.
2. `failed_settlements` = 0, `failed_market_completions` = 0.
3. 시장가 주문 중 PENDING/PARTIAL 잔존 = 0 (시장가 완료 경로가 라이브로 전부 처리됐는가).
4. outbox PENDING 잔존 = 0 (드레인 완료 후).
5. `settlement_batch_fallbacks_total` = 0.

## 판정 기준

**1부**: ① 측정 2의 flush 평균이 512 대비 뚜렷한 여유(포화 해제) ② execution 채널이 만석 고정에서 풀림 ③ 처리량 상승 폭 기록(수치 예단 없음 — DB CPU가 캡이면 상승이 제한될 수 있고, 그 경우 "다음 병목 = DB CPU 확정"이 결론) ④ 병렬 writer 백로그 판정: 새 상한에도 flush 포화 + 채널 만석이면 승격, 아니면 기각 사유 기록.

**2부**: ① 스파이크 전 구간 비-취소 에러율 < 1%(k6 http_req_failed + 커스텀 카운터) ② backend 크래시·재시작 0 ③ 위 정합성 검사 5항목 전부 0 ④ 회복: 스파이크 종료 후 2분 내 p95가 평시 구간 수준(±50%)으로 복귀 — k6 시계열(30초 단위 진행 로그)과 서버 메트릭 샘플로 판정. **미충족 항목도 그대로 기록**(19~21번 관례 — 특히 ③이 하나라도 깨지면 그것이 이 프로젝트 최대의 발견이다).

## 절차

1. 사전 작업 3건 구현·로컬 스모크·커밋(author→reviewer)·푸시.
2. VM 3대 start → db postgres 수동 확인 → 스크립트 2개(멀티심볼·스파이크) load-gen 복사 → 토큰 절차(19번 방식).
3. bench-b22 배포(현 main, 1회) → **기동 로그에서 outbox batch size + shards=1 확인**.
4. 측정 1(env=64) → 리셋 → 측정 2(env=512, 수집 5종) → (조건부 3) → 리셋 → 측정 4(스파이크) → 드레인 대기 → 정합성 검사 5항목.
5. 문서화: `docs/benchmarks/22-...md`(판정 1부 4개 + 2부 4개 명시) + 완료 문서 9번 "남은 것" 갱신 + README(병렬 writer 판정, 급등락 내성 결과를 현황판 9번 행 또는 신규 서술로).
6. VM stop → 커밋(author→reviewer) + 푸시 + CI.
