# B-3 심볼 샤딩 GCP 재측정 계획 (21번 벤치마크)

> **For agentic workers:** 이 계획은 측정 절차다. 순서대로 실행하고, 원본 출력을 보존한 뒤 21번 벤치마크 문서로 정리한다.

**Goal:** B-3(매칭 엔진 심볼 샤딩)의 실효를 같은 세션 A/B로 판정한다. 판정 신호 3개:
1. **샤드별 order 채널이 1024 고정에서 풀리는가** — 20번에서 단일 엔진 채널이 hold 8분 내내 상한에 고정돼 있었다.
2. **1,479 iter/s 캡을 돌파하는가** (20번 측정 3, 단 세션이 다르므로 이번 세션의 A가 기준).
3. **다음 병목 식별** — 엔진 캡이 풀리면 pprof상 주문 생성 경로의 DB 왕복(50%)이 후보. DB VM CPU도 함께 본다.

**측정 대상 커밋:**
- **A (before)**: `632455a` — 파이프라인 전환(`f3d2818`) 직전. ShardedEngine 코드는 있지만 main.go는 아직 단일 엔진(동작상 20번 측정 3과 동일 코드).
- **B (after)**: 현 main(`16da453`) + **`GOEXCHANGE_ENGINE_SHARDS=8`** (심볼 8개 = 샤드 8개, 라운드로빈이라 심볼당 정확히 1샤드).

## 방법론 (17~20번 계승, 위반 금지)

- **같은 세션 A/B**: 서버·DB VM은 측정 도중 stop/start 금지. 20번 수치와 직접 비교 금지 — 이번 세션의 A가 유일한 기준선.
- **매 측정 전 리셋**: TRUNCATE(19번 문서의 9테이블 커맨드) + backend 재시작 + bootstrap loaded=0 확인.
- **부하**: `loadtest/order-submission-multisymbol.js`(리포에 커밋됨, 8심볼) hold 프로파일. load-gen VM에 복사해 사용.
- **B 배포 시 env**: 서버 `.env`(또는 compose)에 `GOEXCHANGE_ENGINE_SHARDS=8` 추가. A는 이 변수를 모르는 코드라 설정돼 있어도 무해하지만, 혼동을 없애려면 A 측정 동안은 빼둔다. `GOEXCHANGE_ENABLE_PPROF=true`는 두 빌드 동일 유지(기존 세션부터 설정돼 있음).
- 기동 로그로 배포 확인: B는 `matching engine sharded: shards=8`이 찍혀야 한다.

## 측정 매트릭스 (기본 2회, 조건부 +1)

| # | 빌드 | 샤드 | 목적 |
|---|---|---|---|
| 1 | A (`632455a`) | 1 (단일 엔진) | 기준선 |
| 2 | B (main) | 8 | B-3 효과 + 판정 신호 3개 수집 |
| 3(조건부) | B (main) | 1 | 측정 2가 A 대비 회귀하거나 이득이 ±5% 이내로 애매할 때만 — 라우터/팬인 오버헤드를 샤딩 병렬화와 분리 |

WS 시나리오는 없음 — B-3의 목표는 REST 처리량 캡이고, WS 상호작용은 19·20번에서 이미 규명했다.

## 측정 2에서 함께 수집 (판정 신호용)

1. **샤드별 order 채널 게이지**: `/metrics`의 `matching_engine_shard_order_channel_length{shard=...}` 8개 + 합산 게이지를 hold 구간 동안 30초 간격 샘플링. 판정: 전 샤드가 1024 고정에서 풀렸는지(간헐 적체는 정상), 샤드 간 불균형 여부(라운드로빈 검증).
2. **pprof CPU 30초** (hold 중반, 20번과 동일 절차) — 다음 병목 식별.
3. **DB VM CPU**: hold 중반에 db VM에서 `top -bn3` 등으로 postgres CPU 점유 샘플 — 처리량이 크게 오르면 DB 포화가 다음 캡일 수 있다.
4. **settlement 지표**: `settlement_batch_size` 분포/fallbacks(관례), `settlement_worker_queue_length`(20번의 불균형 관찰 후속 — 백로그 승격 판단 자료).
5. **load-gen 자원 감시**: 처리량이 크게 오르면 k6가 다시 병목이 될 수 있다(19번 OOM 선례). hold 중 k6 프로세스 CPU/메모리를 확인하고, load-gen이 포화됐다면 측정값은 하한임을 문서에 명시한다.

## 판정 기준

- **B-3 성공**: 측정 2에서 샤드 채널들이 상한 고정에서 풀리고 처리량이 측정 1을 유의미하게(>10%) 상회.
- **채널은 풀렸는데 처리량 정체**: 실패가 아니다 — 직렬화 캡은 제거됐고 병목이 이미 다른 곳(주문 생성 DB 왕복 등)이라는 뜻. pprof·DB CPU로 다음 병목을 지목해 기록하는 것이 이 측정의 세 번째 목적이다.
- **회귀(측정 2 < 측정 1)**: 측정 3(B, 샤드 1)으로 라우터 오버헤드 분리 후 원인 기록. `http_req_failed` > 0이면 원인 파악 전 다음 측정 금지. 예상 밖 결과도 그대로 기록(19·20번 관례).

## 절차

1. VM 3대 start → db postgres 컨테이너 확인 → 스크립트 load-gen 복사 → 토큰 준비(19번 방식, 트랜스크립트 노출 금지).

3. B(main) 배포(+`GOEXCHANGE_ENGINE_SHARDS=8`, 기동 로그 확인) → 리셋 → 측정 2(+수집 5종).
4. (조건 충족 시) B 샤드1 재기동 → 리셋 → 측정 3.
5. 문서화: `docs/benchmarks/21-...md`(기존 컨벤션 + 판정 신호 3개 명시) + 완료 문서 8번 "남은 것" 갱신 + README 8번 행 갱신 + **백로그 승격 판단 기록**(정산 워커 불균형·다음 병목 항목을 근거와 함께 로드맵 후보로).
6. VM 3대 stop → 커밋(author→reviewer) + 푸시 + CI 확인.
